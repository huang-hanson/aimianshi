# 604.MySQL中的MVCC是什么？


## 一、核心定义

MySQL InnoDB 中：
- **聚簇索引（主键索引）：有完整的 MVCC 多版本快照**，存储了行数据的所有历史版本、`trx_id`/`roll_ptr`等隐藏字段，是 MVCC 的核心载体。
- **二级索引（非主键索引）： 没有独立的 MVCC 多版本快照**，二级索引只存储最新版本的数据，不保存历史版本。
- **二级索引的 MVCC 一致性，是通过「回表到聚簇索引」来实现的。**

## 二、原理拆解

### 1. 先搞懂：聚簇索引和二级索引的本质区别

| 索引类型 | 存储内容 | 特点 |
| :--- | :--- | :--- |
| 聚簇索引（主键索引） | 完整的行数据 + 3 个隐藏字段（`trx_id`/`roll_ptr`/`row_id`） | 数据本身就是索引，是 InnoDB 数据存储的核心 |
| 二级索引（非主键索引） | 索引列的值 + 主键 ID | 相当于「索引列 → 主键 ID」的映射表，不存完整行数据 |


### 2. 为什么二级索引不存 MVCC 快照？
- **存储成本爆炸**：如果二级索引也存所有历史版本，每修改一次数据，所有二级索引都要新增历史版本，存储量会翻 N 倍，完全不可行。
- **设计逻辑**：MVCC 的历史版本只存在于「唯一的数据源」—— 聚簇索引中，二级索引只存最新数据，通过回表去聚簇索引拿历史版本，既节省空间，又保证一致性。

### 3.二级索引如何实现 MVCC 一致性？（完整流程）
我们用一个真实场景，走一遍完整流程，秒懂原理：

**场景设定**

- 表结构：`user(id INT PRIMARY KEY, name VARCHAR(20), age INT)`
- 二级索引：`idx_name（name列）`
- 初始数据：`id=1, name='张三', age=18`（聚簇索引存完整数据 + 隐藏字段，二级索引`idx_name`存(`'张三', 1`)）
- 事务 A：`SELECT * FROM user WHERE name='张三'`;（快照读，RR 隔离级别）
- 事务 B：`UPDATE user SET name='李四' WHERE id=1;`（未提交）

**步骤 1：事务 B 修改数据，聚簇索引生成新版本**

事务 B 修改`name='李四'`时：
- 聚簇索引：生成新数据版本`id=1, name='李四', age=18，trx_id=2，roll_ptr`指向旧版本('张三', 18)，形成版本链。
- 二级索引`idx_name`：**`直接删除旧条目('张三', 1)，新增新条目('李四', 1)`**，只存最新版本，不存历史版本。

**步骤 2：事务 A 执行查询，走二级索引idx_name**

事务 A 查询`name='张三'`：
1. 先扫描二级索引`idx_name`，发现**没有'张三'这个条目**（已经被事务 B 删掉了，只剩'李四'）。
2. 但事务 A 的`read view`（一致性视图）规定：事务 B 的修改（`trx_id=2`）对事务 A 不可见。
3. 于是 InnoDB 会**回表到聚簇索引**，通过主键 ID=1，遍历聚簇索引的版本链，找到对事务 A 可见的旧版本`name='张三'`。
4. 最终事务 A 读到`name='张三'`，完美实现 MVCC 一致性，哪怕二级索引已经没有这个条目了。

## 4. 关键补充：二级索引的「当前读」和「快照读」
   
**（1）快照读（普通 SELECT）**
   - 走二级索引时，通过「回表到聚簇索引」，基于聚簇索引的版本链和read view，拿到可见版本，实现一致性。
   - 二级索引本身不存历史版本，只负责定位主键 ID，一致性由聚簇索引保证。
   
**（2）当前读（SELECT ... FOR UPDATE / UPDATE / DELETE）**
   - 走二级索引时，直接读取二级索引的最新版本，加 Next-Key Lock（间隙锁 + 行锁），防止幻读。
   - 因为当前读要读最新数据，不需要历史版本，直接用二级索引的最新数据即可。
