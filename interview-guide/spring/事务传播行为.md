# [Spring] 事务传播行为详解

## 一、题目分析

### 考察点
- Spring 事务传播行为的定义
- 7 种传播行为的含义和使用场景
- REQUIRED 和 REQUIRES_NEW 的区别
- 事务传播行为的实际应用
- 嵌套事务的理解

### 难度等级
⭐⭐⭐⭐⭐ (5/5 星) - 高频核心难题

### 适用岗位
- 中级/高级 Java 开发工程师
- Spring 框架相关岗位必考题

---

## 二、核心答案

**一句话回答：**
> Spring 事务传播行为定义了**多个事务方法相互调用时，事务如何传播**。Spring 定义了**7 种传播行为**：REQUIRED（默认，加入现有事务或新建）、REQUIRES_NEW（总是新建事务）、SUPPORTS（支持事务或无事务）、NOT_SUPPORTED（不支持事务）、MANDATORY（必须在事务中）、NEVER（不能在事务中）、NESTED（嵌套事务）。

## 三、深度剖析

### 3.1 7 种事务传播行为

```
┌─────────────────────────────────────────────────────────┐
│          Spring 7 种事务传播行为                          │
│                                                          │
│  Propagation 枚举（TransactionDefinition 接口）：         │
│                                                          │
│  1. REQUIRED（默认）                                      │
│     - 当前有事务：加入现有事务                            │
│     - 当前无事务：新建事务                               │
│                                                          │
│  2. REQUIRES_NEW                                        │
│     - 当前有事务：挂起现有事务，新建事务                 │
│     - 当前无事务：新建事务                               │
│                                                          │
│  3. SUPPORTS                                            │
│     - 当前有事务：加入现有事务                            │
│     - 当前无事务：以非事务方式执行                       │
│                                                          │
│  4. NOT_SUPPORTED                                       │
│     - 当前有事务：挂起现有事务，以非事务方式执行         │
│     - 当前无事务：以非事务方式执行                       │
│                                                          │
│  5. MANDATORY                                           │
│     - 当前有事务：加入现有事务                            │
│     - 当前无事务：抛异常（IllegalTransactionStateException）│
│                                                          │
│  6. NEVER                                               │
│     - 当前有事务：抛异常（IllegalTransactionStateException）│
│     - 当前无事务：以非事务方式执行                       │
│                                                          │
│  7. NESTED                                              │
│     - 当前有事务：创建嵌套事务（保存点）                 │
│     - 当前无事务：新建事务                               │
└─────────────────────────────────────────────────────────┘
```

### 3.3 REQUIRED vs REQUIRES_NEW 对比

```
┌─────────────────────────────────────────────────────────┐
│          REQUIRED vs REQUIRES_NEW                        │
│                                                          │
│  场景：ServiceA.methodA() 调用 ServiceB.methodB()          │
│                                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │  情况 1：methodA 有事务，methodB=REQUIRED           │    │
│  │                                                  │    │
│  │  ServiceA.methodA(@Transactional)               │    │
│  │     │                                            │    │
│  │     └─> ServiceB.methodB(@Transactional)        │    │
│  │          ↓                                       │    │
│  │          加入 methodA 的同一事务                    │    │
│  │                                                  │    │
│  │  结果：                                           │    │
│  │  - methodB 回滚 → methodA 也回滚                     │    │
│  │  - methodA 回滚 → methodB 也回滚                     │    │
│  │  - 同一个事务，一荣俱荣，一损俱损                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │  情况 2：methodA 有事务，methodB=REQUIRES_NEW     │    │
│  │                                                  │    │
│  │  ServiceA.methodA(@Transactional)               │    │
│  │     │                                            │    │
│  │     └─> ServiceB.methodB(@Transactional(propagation=REQUIRES_NEW)) │
│  │          ↓                                       │    │
│  │          挂起 methodA 事务，新建独立事务             │    │
│  │                                                  │    │
│  │  结果：                                           │    │
│  │  - methodB 回滚 → methodA 不一定回滚（可捕获异常）   │    │
│  │  - methodA 回滚 → methodB 已提交则不回滚            │    │
│  │  - 两个独立事务，互不影响                          │    │
│  └─────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
```

### 3.4 嵌套事务（NESTED）详解

```
┌─────────────────────────────────────────────────────────┐
│          NESTED 嵌套事务                                 │
│                                                          │
│  特点：                                                   │
│  - 基于数据库保存点（Savepoint）实现                     │
│  - 嵌套事务可以独立回滚，不影响外层事务                  │
│  - 外层事务回滚，嵌套事务也会回滚                        │
│  - 仅支持 JDBC 3.0+，需要 DataSourceTransactionManager    │
│                                                          │
│  执行流程：                                               │
│  ┌─────────────────────────────────────────────────┐    │
│  │  ServiceA.methodA(@Transactional)               │    │
│  │     │                                            │    │
│  │     ├─> 开启事务 T1                               │    │
│  │     │                                            │    │
│  │     ├─> ServiceB.methodB(NESTED)                │    │
│  │     │    │                                        │    │
│  │     │    └─> 创建保存点 SP1                        │    │
│  │     │         │                                    │    │
│  │     │         ├─> 业务逻辑                         │    │
│  │     │         │                                    │    │
│  │     │         ├─> 异常？                          │    │
│  │     │         │   ├─> 是：回滚到 SP1（不影响 T1）   │    │
│  │     │         │   └─> 否：继续                     │    │
│  │     │                                            │    │
│  │     ├─> ServiceC.methodC(NESTED)                │    │
│  │     │    └─> 创建保存点 SP2 ...                   │    │
│  │     │                                            │    │
│  │     └─> 提交/回滚 T1                              │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  REQUIRED vs NESTED：                                    │
│  - REQUIRED：完全加入外层事务，一损俱损                  │
│  - NESTED：可以独立回滚，但外层回滚会影响嵌套            │
└─────────────────────────────────────────────────────────┘
```

## 四、代码示例

### 4.1 REQUIRED（默认）示例

```java
import org.springframework.stereotype.*;
import org.springframework.transaction.annotation.*;

@Service
public class RequiredService {

    @Autowired
    private RequiredService self;

    @Autowired
    private OtherService otherService;

    // 方法 A：REQUIRED（默认）
    @Transactional
    public void methodA() {
        System.out.println("methodA：开始");
        // 执行数据库操作 1
        insertData("A");
        // 调用方法 B（REQUIRED）
        self.methodB();
        System.out.println("methodA：结束");
    }

    // 方法 B：REQUIRED
    @Transactional(propagation = Propagation.REQUIRED)
    public void methodB() {
        System.out.println("methodB：开始");
        insertData("B");
        // 如果这里抛异常，methodA 和 methodB 都回滚
        // throw new RuntimeException();
        System.out.println("methodB：结束");
    }

    // 方法 C：REQUIRED，调用外部服务
    @Transactional
    public void methodC() {
        System.out.println("methodC：开始");
        insertData("C");
        // 调用外部服务（REQUIRED）
        otherService.otherMethod();
        System.out.println("methodC：结束");
    }

    private void insertData(String data) {
        // 数据库操作
    }
}
```

**测试结果：**
```
场景 1：methodB 正常执行
- methodA 和 methodB 在同一事务
- 一起提交

场景 2：methodB 抛异常
- methodA 和 methodB 都回滚
- 因为它们在同一个事务中
```

### 4.2 REQUIRES_NEW 示例

```java
@Service
public class RequiresNewService {

    @Autowired
    private RequiresNewService self;

    @Autowired
    private LogService logService;

    // 方法 A：REQUIRED
    @Transactional
    public void methodA() {
        System.out.println("methodA：开始");
        insertData("A");

        try {
            // 调用方法 B（REQUIRES_NEW）
            self.methodB();
        } catch (Exception e) {
            // 捕获异常，methodA 不回滚
            System.out.println("捕获异常，methodA 继续");
        }

        System.out.println("methodA：结束");
    }

    // 方法 B：REQUIRES_NEW
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void methodB() {
        System.out.println("methodB：开始");
        insertData("B");
        // 抛异常，methodB 回滚，但不影响 methodA
        throw new RuntimeException("methodB 异常");
    }

    // 记录日志（总是新建事务）
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void log(String message) {
        System.out.println("记录日志：" + message);
        logDao.insert(message);
        // 即使外层回滚，日志也提交
    }
}
```

**测试结果：**
```
场景 1：methodB 抛异常，methodA 捕获
- methodB 回滚（独立事务）
- methodA 提交（不受影响）

场景 2：记录日志
- 外层事务回滚
- 日志仍然提交（REQUIRES_NEW）
```

### 4.3 NESTED 示例

```java
@Service
public class NestedService {

    @Autowired
    private NestedService self;

    // 方法 A：REQUIRED
    @Transactional
    public void methodA() {
        System.out.println("methodA：开始");
        insertData("A");

        try {
            // 调用方法 B（NESTED）
            self.methodB();
        } catch (Exception e) {
            // 捕获异常，methodA 不回滚
            System.out.println("捕获异常，methodA 继续");
        }

        System.out.println("methodA：结束");
    }

    // 方法 B：NESTED
    @Transactional(propagation = Propagation.NESTED)
    public void methodB() {
        System.out.println("methodB：开始");
        insertData("B");
        // 抛异常，methodB 回滚到保存点，不影响 methodA
        throw new RuntimeException("methodB 异常");
    }

    // 方法 C：NESTED，多个嵌套事务
    @Transactional
    public void methodC() {
        System.out.println("methodC：开始");
        insertData("C");

        try {
            self.methodD();
        } catch (Exception e) {
            System.out.println("methodD 失败，继续执行");
        }

        self.methodE();  // 继续执行
        System.out.println("methodC：结束");
    }

    @Transactional(propagation = Propagation.NESTED)
    public void methodD() {
        System.out.println("methodD：开始");
        insertData("D");
        throw new RuntimeException("methodD 异常");
    }

    @Transactional(propagation = Propagation.NESTED)
    public void methodE() {
        System.out.println("methodE：开始");
        insertData("E");
        System.out.println("methodE：结束");
    }
}
```

**测试结果：**
```
场景 1：methodB 抛异常，methodA 捕获
- methodB 回滚到保存点
- methodA 继续执行并提交

场景 2：methodD 抛异常，methodC 捕获
- methodD 回滚到保存点
- methodE 继续执行
- methodC 提交
```

### 4.4 其他传播行为示例

```java
@Service
public class OtherPropagationService {

    // SUPPORTS：支持事务或无事务
    @Transactional(propagation = Propagation.SUPPORTS)
    public void supportsMethod() {
        System.out.println("SUPPORTS：有事务就加入，无事务就非事务执行");
        // 适合查询操作
        queryData();
    }

    // NOT_SUPPORTED：不支持事务
    @Transactional(propagation = Propagation.NOT_SUPPORTED)
    public void notSupportedMethod() {
        System.out.println("NOT_SUPPORTED：挂起事务，非事务执行");
        // 适合批量操作、耗时操作
        batchInsert();
    }

    // MANDATORY：必须在事务中
    @Transactional(propagation = Propagation.MANDATORY)
    public void mandatoryMethod() {
        System.out.println("MANDATORY：必须在事务中执行");
        // 如果外层没有事务，抛异常
        insertData("MANDATORY");
    }

    // NEVER：不能在事务中
    @Transactional(propagation = Propagation.NEVER)
    public void neverMethod() {
        System.out.println("NEVER：不能在事务中执行");
        // 如果外层有事务，抛异常
        nonTransactionalOperation();
    }

    private void queryData() { }
    private void batchInsert() { }
    private void insertData(String s) { }
    private void nonTransactionalOperation() { }
}
```

## 五、常见面试题扩展

### 扩展 1：REQUIRED 和 REQUIRES_NEW 的区别？

**答：**

| 对比项 | REQUIRED | REQUIRES_NEW |
|--------|----------|--------------|
| **有事务时** | 加入现有事务 | 挂起现有事务，新建事务 |
| **无事务时** | 新建事务 | 新建事务 |
| **回滚影响** | 一起回滚 | 独立回滚，互不影响 |
| **锁持有** | 同一事务，锁持有到外层提交 | 独立事务，锁先释放 |
| **性能** | 较高（一个事务） | 较低（两个事务） |
| **使用场景** | 默认选择 | 日志记录、独立操作 |

**示例：**
```java
// REQUIRED：订单和订单项一起提交/回滚
@Transactional
public void createOrder(Order order) {
    orderDao.insert(order);
    for (OrderItem item : order.getItems()) {
        orderItemDao.insert(item);  // REQUIRED，一起回滚
    }
}

// REQUIRES_NEW：日志总是提交
@Transactional
public void businessMethod() {
    try {
        // 业务逻辑
    } catch (Exception e) {
        logService.log("操作失败");  // REQUIRES_NEW，总是提交
        throw e;
    }
}
```

### 扩展 2：NESTED 和 REQUIRES_NEW 的区别？

**答：**

| 对比项 | NESTED | REQUIRES_NEW |
|--------|--------|--------------|
| **实现方式** | 保存点（Savepoint） | 新建事务 |
| **有事务时** | 创建嵌套事务 | 挂起外层，新建事务 |
| **内层回滚** | 回滚到保存点，不影响外层 | 内层独立回滚 |
| **外层回滚** | 内层也回滚 | 内层已提交则不回滚 |
| **数据库要求** | 支持保存点（JDBC 3.0+） | 无特殊要求 |
| **性能** | 较高（一个事务） | 较低（两个事务） |

**选择建议：**
- 需要内层完全独立：用 REQUIRES_NEW
- 内层可以受外层影响：用 NESTED
- 性能敏感：用 NESTED

### 扩展 3：事务传播行为失效的场景？

**答：**

```
1. 同类方法调用（自调用）
   @Service
   public class MyService {
       @Transactional
       public void methodA() {
           methodB();  // 失效！直接调用，不经过代理
       }

       @Transactional(propagation = Propagation.REQUIRES_NEW)
       public void methodB() { }
   }

   解决：
   - 注入自身：@Autowired private MyService self;
   - 使用 AopContext：((MyService) AopContext.currentProxy()).methodB();

2. 非 public 方法
   @Transactional
   protected void method() { }  // 失效！只支持 public

3. 异常被捕获
   @Transactional
   public void method() {
       try {
           // 抛异常
       } catch (Exception e) {
           // 吞掉异常，不回滚
       }
   }

4. 异常类型不匹配
   @Transactional(rollbackFor = RuntimeException.class)
   public void method() throws Exception {
       throw new Exception();  // 不回滚！默认只回滚 RuntimeException
   }

5. 数据库不支持
   - NESTED 需要数据库支持保存点
   - MySQL InnoDB 支持，MyISAM 不支持
```

### 扩展 4：如何配置事务传播行为？

**答：**

```java
// 1. @Transactional 注解（推荐）
@Transactional(propagation = Propagation.REQUIRED)
public void method() { }

// 2. XML 配置
<tx:advice id="txAdvice" transaction-manager="txManager">
    <tx:attributes>
        <tx:method name="create*" propagation="REQUIRED"/>
        <tx:method name="update*" propagation="REQUIRED"/>
        <tx:method name="delete*" propagation="REQUIRED"/>
        <tx:method name="query*" propagation="SUPPORTS" read-only="true"/>
        <tx:method name="log*" propagation="REQUIRES_NEW"/>
    </tx:attributes>
</tx:advice>

// 3. 编程式配置
TransactionTemplate template = new TransactionTemplate(txManager);
template.setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRED);
template.execute(status -> {
    // 业务逻辑
    return null;
});
```

### 扩展 5：事务传播行为在实际项目中的应用？

**答：**

```
场景 1：订单创建（REQUIRED）
@Transactional
public void createOrder(Order order) {
    orderDao.insert(order);
    for (OrderItem item : order.getItems()) {
        orderItemDao.insert(item);  // REQUIRED，一起提交/回滚
    }
    inventoryService.reduceStock(order);  // REQUIRED，库存一起扣减
}

场景 2：操作日志（REQUIRES_NEW）
@Transactional
public void businessOperation() {
    try {
        // 业务逻辑
    } catch (Exception e) {
        logService.log("操作失败");  // REQUIRES_NEW，总是提交
        throw e;
    }
}

场景 3：批量处理（NOT_SUPPORTED）
@Transactional
public void batchProcess() {
    for (Data data : dataList) {
        processSingle(data);  // NOT_SUPPORTED，挂起事务，提高性能
    }
}

场景 4：查询操作（SUPPORTS）
@Transactional(propagation = Propagation.SUPPORTS, read-only = true)
public List<Data> queryData() {
    return dataDao.findAll();
}

场景 5：内部方法必须在事务中（MANDATORY）
@Transactional(propagation = Propagation.MANDATORY)
public void internalMethod() {
    // 必须由外层事务调用
}
```

## 六、面试答题话术

**面试官问：请说说 Spring 事务的传播行为？**

**标准回答：**

> Spring 定义了 7 种事务传播行为，用于控制多个事务方法相互调用时事务如何传播：
>
> 1. **REQUIRED**（默认）：有事务就加入，无事务就新建。最常用。
> 2. **REQUIRES_NEW**：总是新建事务，有事务就挂起。适合日志记录。
> 3. **SUPPORTS**：有事务就加入，无事务就非事务执行。适合查询。
> 4. **NOT_SUPPORTED**：有事务就挂起，非事务执行。适合批量操作。
> 5. **MANDATORY**：必须在事务中，否则抛异常。
> 6. **NEVER**：不能在事务中，否则抛异常。
> 7. **NESTED**：有事务就创建嵌套事务（保存点），无事务就新建。
>
> 最常用的是 REQUIRED 和 REQUIRES_NEW，前者加入现有事务，后者总是新建独立事务。

**加分回答：**

> 补充几点：
> 1. REQUIRED 和 REQUIRES_NEW 的区别：REQUIRED 回滚时一起回滚，REQUIRES_NEW 可以独立回滚
> 2. NESTED 基于数据库保存点实现，内层回滚不影响外层，但外层回滚会影响内层
> 3. 同类方法调用时事务会失效，因为不经过代理，需要注入自身或使用 AopContext
> 4. 实际开发中，业务方法用 REQUIRED，日志用 REQUIRES_NEW，查询用 SUPPORTS
> 5. 事务传播行为只在通过代理调用时才生效，直接调用不生效
> 6. MANDATORY 和 NEVER 用于强制约束调用环境，防止误用

## 七、速记表格

| 传播行为 | 有事务 | 无事务 | 使用场景 |
|----------|--------|--------|----------|
| **REQUIRED** | 加入 | 新建 | 默认，业务方法 |
| **REQUIRES_NEW** | 挂起，新建 | 新建 | 日志、独立操作 |
| **SUPPORTS** | 加入 | 非事务 | 查询操作 |
| **NOT_SUPPORTED** | 挂起 | 非事务 | 批量操作 |
| **MANDATORY** | 加入 | 抛异常 | 强制事务环境 |
| **NEVER** | 抛异常 | 非事务 | 强制非事务 |
| **NESTED** | 嵌套事务 | 新建 | 部分回滚场景 |
