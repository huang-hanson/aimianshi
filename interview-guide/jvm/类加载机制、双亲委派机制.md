# [JVM] 类加载机制、双亲委派机制

## 一、题目分析

### 考察点
- JVM 类加载过程（加载、验证、准备、解析、初始化）
- 类加载器的层次结构
- 双亲委派模型的原理
- 双亲委派的优缺点及破坏场景

### 难度等级
⭐⭐⭐⭐ (4/5 星) - 高频核心题

### 适用岗位
- 中级/高级 Java 开发工程师
- JVM 调优、框架开发相关岗位必考题

## 二、核心答案

**一句话回答：**
> Java 类加载机制分为**加载、验证、准备、解析、初始化**五个阶段。**双亲委派模型**规定：类加载器收到请求后，先委托给父加载器处理，只有父加载器无法加载时，才自己尝试加载。这样可以保证核心类的安全性和一致性。

## 三、深度剖析

### 3.1 类加载过程详解

```
┌─────────────────────────────────────────────────────────┐
│              类加载五阶段流程                            │
│                                                          │
│  加载 → 验证 → 准备 → 解析 → 初始化                       │
│   │      │      │      │      │                         │
│   │      │      │      │      └─> 执行 <clinit>()       │
│   │      │      │      └─> 符号引用转直接引用             │
│   │      │      └─> 分配内存、设默认值                    │
│   │      └─> 校验字节码、安全性                          │
│   └─> 读取字节流、生成 Class 对象                          │
│                                                          │
│  其中：验证、准备、解析 三个阶段合称「连接」                │
└─────────────────────────────────────────────────────────┘
```

### 3.2 类加载五阶段详解

| 阶段 | 主要任务 | 关键细节 |
|------|----------|----------|
| **加载** | 读取字节流，生成 Class 对象 | 可从 class 文件、jar 包、网络、动态生成 |
| **验证** | 确保字节码安全、符合规范 | 文件格式验证、元数据验证、字节码验证 |
| **准备** | 分配内存，设置默认值 | `static` 变量设默认值，**不执行赋值语句** |
| **解析** | 符号引用转直接引用 | 解析类、字段、方法、接口调用等 |
| **初始化** | 执行 `<clinit>()` 方法 | 执行静态代码块、`static` 赋值语句 |

### 3.3 准备阶段 vs 初始化阶段

```java
public class Test {
    // 准备阶段：value = 0（默认值）
    // 初始化阶段：value = 123（执行赋值）
    public static int value = 123;

    // 准备阶段：constant = 0（默认值）
    // 初始化阶段：constant = 456（执行赋值）
    public static final int constant = 456;

    // 准备阶段：不分配内存（final static 常量）
    // 初始化阶段：不执行（编译期已放入常量池）
    public static final int COMPILE_CONSTANT = 789;

    // 静态代码块（初始化阶段执行）
    static {
        System.out.println("静态代码块执行");
    }
}
```

**关键区别：**
- **准备阶段**：只分配内存，设为**默认值**（0、null、false）
- **初始化阶段**：执行**赋值语句**和**静态代码块**

│           │ 委托                                         │

│  ┌─────────────────┐                                    │

### 3.5 双亲委派模型

```
┌─────────────────────────────────────────────────────────┐
│              双亲委派流程                                │
│                                                          │
│  自定义 ClassLoader 收到加载请求                          │
│           │                                              │
│           ↓ 先委托给父加载器                              │
│  Application ClassLoader                                 │
│           │                                              │
│           ↓ 先委托给父加载器                              │
│  Extension ClassLoader                                   │
│           │                                              │
│           ↓ 先委托给父加载器                              │
│  Bootstrap ClassLoader                                   │
│           │                                              │
│           ↓ 无法加载（核心类）                            │
│           │                                              │
│           ← 逐级返回，自己尝试加载                        │
│                                                          │
│  核心思想：优先委托给父加载器，自己只加载父加载器无法加载的类 │
└─────────────────────────────────────────────────────────┘
```

### 3.6 双亲委派源码分析

```java
// ClassLoader 的 loadClass 方法（双亲委派核心）
protected Class<?> loadClass(String name, boolean resolve)
    throws ClassNotFoundException {

    synchronized (getClassLoadingLock(name)) {
        // 1. 先检查是否已加载
        Class<?> c = findLoadedClass(name);

        if (c == null) {
            try {
                // 2. 有父加载器，委托给父加载器
                if (parent != null) {
                    c = parent.loadClass(name, false);
                }
                // 3. 无父加载器，使用启动类加载器
                else {
                    c = findBootstrapClassOrNull(name);
                }
            } catch (ClassNotFoundException e) {
                // 父加载器无法加载，捕获异常
            }

            // 4. 父加载器无法加载，自己尝试加载
            if (c == null) {
                c = findClass(name);  // 子类重写此方法
            }
        }

        if (resolve) {
            resolveClass(c);  // 链接
        }

        return c;
    }
}
```

**关键点：**
1. 先检查是否已加载
2. 有父加载器先委托
3. 父加载器无法加载才自己加载
4. `findClass()` 方法由子类重写

### 3.7 各类加载器加载范围

| 类加载器 | 加载范围 | 示例 |
|----------|----------|------|
| Bootstrap | 核心类库 | `java.lang.*`、`java.util.*` |
| Extension | 扩展类库 | `jre/lib/ext` 目录下的 jar |
| Application | 应用类 | classpath 下的 class、jar |
| Custom | 自定义类 | 网络、数据库、动态生成 |

### 4.2 自定义类加载器

```java
import java.io.*;
import java.nio.file.*;

// 自定义类加载器
public class MyClassLoader extends ClassLoader {
    private String classPath;

    public MyClassLoader(String classPath) {
        this.classPath = classPath;
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        try {
            // 1. 读取 class 文件字节流
            byte[] data = loadClassData(name);
            // 2. 调用 defineClass 生成 Class 对象
            return defineClass(name, data, 0, data.length);
        } catch (IOException e) {
            throw new ClassNotFoundException(name);
        }
    }

    private byte[] loadClassData(String className) throws IOException {
        // 将类名转换为文件路径
        String path = classPath + File.separator +
                      className.replace('.', File.separatorChar) + ".class";

        // 读取文件
        FileInputStream fis = new FileInputStream(path);
        ByteArrayOutputStream baos = new ByteArrayOutputStream();

        byte[] buffer = new byte[1024];
        int len;
        while ((len = fis.read(buffer)) != -1) {
            baos.write(buffer, 0, len);
        }

        fis.close();
        return baos.toByteArray();
    }

    public static void main(String[] args) throws Exception {
        // 使用自定义类加载器
        MyClassLoader loader = new MyClassLoader("D:/classes");
        Class<?> clazz = loader.loadClass("com.example.MyClass");
        Object obj = clazz.newInstance();

        System.out.println("类加载器：" + clazz.getClassLoader());
    }
}
```

### 4.3 破坏双亲委派示例

```java
// 1. 重写 loadClass 方法，不委托给父加载器
public class NonDelegatingClassLoader extends ClassLoader {

    @Override
    protected Class<?> loadClass(String name, boolean resolve)
            throws ClassNotFoundException {
        synchronized (getClassLoadingLock(name)) {
            // 不委托给父加载器，直接自己加载
            Class<?> c = findLoadedClass(name);
            if (c == null) {
                c = findClass(name);  // 直接自己加载
            }
            if (resolve) {
                resolveClass(c);
            }
            return c;
        }
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        // 实现自己的加载逻辑
        // ...
        return super.findClass(name);
    }
}

// 2. 线程上下文类加载器（Thread Context ClassLoader）
public class TCCLExample {
    public static void main(String[] args) {
        // 获取当前线程的上下文类加载器
        ClassLoader tccl = Thread.currentThread().getContextClassLoader();
        System.out.println("TCCL: " + tccl);  // AppClassLoader

        // SPI 机制中常用（如 JDBC、JNDI）
        // 父加载器加载的代码可以通过 TCCL 使用子加载器加载的类
    }
}
```

## 五、常见面试题扩展

### 扩展 1：双亲委派模型有什么好处？

**答：** 主要有两个好处：

```
1. 安全性
   - 防止核心 API 被篡改
   - 例如：自定义 java.lang.String 不会被加载
   - 因为 Bootstrap 加载器会优先加载 rt.jar 中的 String

2. 避免重复加载
   - 同一个类只会被加载一次
   - 保证类的唯一性
   - 例如：ArrayList 只会被 Bootstrap 加载器加载一次
```

### 扩展 2：如何破坏双亲委派模型？

**答：** 常见破坏场景：

```
1. 重写 loadClass 方法
   - 不委托给父加载器，直接自己加载
   - 适用于热部署、模块隔离

2. 线程上下文类加载器（TCCL）
   - JDBC SPI、JNDI、JAXB 等使用
   - 父加载器代码通过 TCCL 使用子加载器加载的类

3. OSGi 模块化
   - 每个模块有自己的类加载器
   - 模块间类加载器是平级关系，不是父子

4. 热部署框架
   - Tomcat、Spring Boot DevTools
   - 实现应用热更新、隔离
```

### 扩展 3：Tomcat 是如何破坏双亲委派的？

**答：** Tomcat 的类加载器结构：

```
Bootstrap ClassLoader
    ↓
Extension ClassLoader
    ↓
Application ClassLoader
    ↓
Common ClassLoader（Tomcat 公共类）
    ↓
    ├─> WebApp ClassLoader A（应用 A）
    ├─> WebApp ClassLoader B（应用 B）
    └─> WebApp ClassLoader C（应用 C）

特点：
1. WebApp 加载器优先加载 webapp 自己的类
2. 特定包（如 servlet）仍委托给父加载器
3. 不同应用的类加载器隔离，可以加载不同版本的类
```

### 扩展 4：什么是 SPI 机制？为什么需要破坏双亲委派？

**答：**

```java
// SPI（Service Provider Interface）机制
// JDBC 驱动加载示例

// 1. 传统方式（破坏双亲委派）
Class.forName("com.mysql.jdbc.Driver");

// 2. SPI 方式（Java 6+）
// META-INF/services/java.sql.Driver 文件中指定实现类
// 自动加载所有驱动实现

// 为什么需要破坏双亲委派？
// Bootstrap 加载器加载的 java.sql.Driver 需要加载
// Application 加载器加载的 MySQL 驱动实现类
// 通过 Thread Context ClassLoader 实现

DriverManager.getConnection(url, user, password);
// DriverManager 使用 TCCL 加载具体的驱动实现
```

### 扩展 5：类加载的时机是什么？

**答：** 以下情况会触发类加载：

```
1. 主动引用（会触发加载）
   - 创建对象（new、反射）
   - 访问静态字段
   - 调用静态方法
   - 子类初始化时（父类先初始化）
   - 启动类（main 方法所在类）

2. 被动引用（不会触发加载）
   - 通过子类引用父类静态字段
   - 定义数组（如 String[]）
   - 引用常量（final static）
```

**代码示例：**
```java
// 1. 主动引用
new MyClass();                    // 会加载
MyClass.staticField;              // 会加载
MyClass.staticMethod();           // 会加载

// 2. 被动引用
SubClass.staticField;             // 不会加载父类（只初始化）
String[] arr = new String[10];    // 不会加载 String 类
System.out.println(MyClass.CONST); // 不会加载（常量）
```

## 六、面试答题话术

**面试官问：请说说 Java 的类加载机制和双亲委派模型？**

**标准回答：**

> Java 类加载机制分为五个阶段：**加载、验证、准备、解析、初始化**。
>
> 1. **加载**：读取字节流，生成 Class 对象
> 2. **验证**：确保字节码安全
> 3. **准备**：分配内存，设默认值
> 4. **解析**：符号引用转直接引用
> 5. **初始化**：执行静态代码块和赋值语句
>
> **双亲委派模型**是指：类加载器收到请求后，先委托给父加载器处理，只有父加载器无法加载时，才自己尝试加载。
>
> 类加载器分为：Bootstrap、Extension、Application 和自定义加载器。双亲委派的好处是保证核心类的安全性和一致性，防止核心 API 被篡改。

**加分回答：**

> 补充几点：
> 1. 准备阶段只设默认值，不执行赋值语句；初始化阶段才执行赋值和静态代码块
> 2. Bootstrap 加载器是 C++ 实现的，返回 null；其他加载器是 Java 实现的
> 3. 双亲委派可以被破坏，常见场景有：SPI 机制（JDBC）、OSGi 模块化、Tomcat 热部署
> 4. SPI 机制通过线程上下文类加载器（TCCL）实现，父加载器加载的代码可以使用子加载器加载的类
> 5. 主动引用会触发类加载（如 new 对象、访问静态成员），被动引用不会（如数组、常量）
