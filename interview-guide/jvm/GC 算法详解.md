# [JVM] GC 算法详解

## 一、题目分析

### 考察点
- 垃圾回收的基本概念
- 判断对象存活的方法
- 经典 GC 算法原理
- GC 分代理论
- 垃圾回收器的实现

### 难度等级
⭐⭐⭐⭐⭐ (5/5 星) - 高频核心难题

### 适用岗位
- 中级/高级 Java 开发工程师
- JVM 调优相关岗位必考题

---

## 二、核心答案

**一句话回答：**
> GC（Garbage Collection）是 JVM 自动内存管理机制。核心流程：**判断对象存活**（可达性分析）→ **标记可回收对象** → **执行回收算法**（标记清除/标记复制/标记整理）。JVM 采用**分代回收**思想，新生代用**复制算法**，老年代用**标记整理算法**。

## 三、深度剖析

### 3.1 GC 整体流程

```
┌─────────────────────────────────────────────────────────┐
│              GC 垃圾回收流程                              │
│                                                          │
│  1. 判断对象是否存活                                      │
│     └─> 引用计数法（已淘汰）                              │
│     └─> 可达性分析（GC Roots）                            │
│                                                          │
│  2. 标记可回收对象                                        │
│     └─> 从 GC Roots 向下遍历                               │
│     └─> 未被引用的对象标记为可回收                        │
│                                                          │
│  3. 执行回收算法                                          │
│     └─> 标记 - 清除（Mark-Sweep）                         │
│     └─> 标记 - 复制（Copying）                            │
│     └─> 标记 - 整理（Mark-Compact）                       │
│                                                          │
│  4. 分代回收                                              │
│     └─> 新生代：Minor GC（复制算法）                      │
│     └─> 老年代：Major GC/Full GC（标记整理）              │
│     └─> 混合回收：Mixed GC（G1）                          │
└─────────────────────────────────────────────────────────┘
```

### 3.2 判断对象存活的方法

```
┌─────────────────────────────────────────────────────────┐
│  方法 1：引用计数法（Reference Counting）                  │
│                                                          │
│  原理：                                                   │
│  - 每个对象有一个引用计数器                              │
│  - 被引用时 +1，引用失效时 -1                            │
│  - 计数器为 0 时可回收                                    │
│                                                          │
│  优点：                                                   │
│  - 简单高效                                              │
│  - 对象死亡立即可知                                      │
│                                                          │
│  缺点：                                                   │
│  - 无法解决循环引用问题                                  │
│  ┌─────────────────────────────────────────────────┐    │
│  │  对象 A → 引用 → 对象 B                           │    │
│  │     ↑          │                                 │    │
│  │     └──────────┘                                 │    │
│  │  A 和 B 互相引用，但实际已不可达，无法回收！         │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  应用：Python、PHP 使用引用计数，Java 未采用               │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  方法 2：可达性分析（Reachability Analysis）              │
│                                                          │
│  原理：                                                   │
│  - 从 GC Roots 对象向下遍历                               │
│  - 形成引用链（Reference Chain）                         │
│  - 未被任何引用链到达的对象可回收                        │
│                                                          │
│  GC Roots 包括：                                          │
│  - 虚拟机栈（栈帧中的局部变量）                          │
│  - 方法区中的静态变量                                    │
│  - 方法区中的常量                                        │
│  - 本地方法栈中的 JNI 引用                               │
│  - JVM 内部引用（类加载器、异常对象等）                    │
│  - 被同步锁持有的对象                                    │
│                                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │           GC Roots                               │    │
│  │         /    |    \                              │    │
│  │       Obj1  Obj2  Obj3                           │    │
│  │       /  \    |                                  │    │
│  │     Obj4 Obj5 Obj6                              │    │
│  │            \                                     │    │
│  │            Obj7 (可回收！)                        │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  Java 采用此方法！                                        │
└─────────────────────────────────────────────────────────┘
```

### 3.3 经典 GC 算法详解

```
┌─────────────────────────────────────────────────────────┐
│  算法 1：标记 - 清除（Mark-Sweep）                        │
│                                                          │
│  执行过程：                                               │
│  1. 标记阶段：从 GC Roots 遍历，标记所有存活对象            │
│  2. 清除阶段：回收未标记的对象                           │
│                                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │  堆内存： [存活][回收][存活][回收][回收][存活]     │    │
│  │            ↓ 标记后                               │    │
│  │           [存活][    ][存活][    ][    ][存活]     │    │
│  │            ↓ 清除后                               │    │
│  │           [存活][空闲][存活][空闲][空闲][存活]     │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  优点：                                                   │
│  - 简单，不需要移动对象                                  │
│                                                          │
│  缺点：                                                   │
│  - 效率不高（标记和清除两个过程）                        │
│  - 内存碎片问题（空闲内存不连续）                        │
│  - 分配大对象时可能触发 GC（找不到连续空间）              │
│                                                          │
│  应用：CMS 收集器的老年代回收                             │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  算法 2：标记 - 复制（Copying）                           │
│                                                          │
│  执行过程：                                               │
│  1. 将内存分为两块（From/To）                            │
│  2. 标记存活对象，复制到 To 区                            │
│  3. 清空 From 区                                         │
│  4. 交换 From/To 角色                                    │
│                                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │  From 区：[存活][回收][存活][回收]                 │    │
│  │           ↓ 复制存活对象                          │    │
│  │  To 区：  [存活][存活][    ][    ]                │    │
│  │           ↓ 清空 From                             │    │
│  │  From 区：[    ][    ][    ][    ] (空闲)          │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  优点：                                                   │
│  - 高效（一次遍历完成标记 + 复制）                        │
│  - 无内存碎片（内存连续）                                │
│  - 分配简单（指针碰撞）                                  │
│                                                          │
│  缺点：                                                   │
│  - 内存利用率低（只有 50% 可用）                           │
│  - 对象存活率高时效率低                                  │
│                                                          │
│  应用：新生代 GC（ParNew、Parallel Scavenge）             │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  算法 3：标记 - 整理（Mark-Compact）                      │
│                                                          │
│  执行过程：                                               │
│  1. 标记阶段：从 GC Roots 遍历，标记所有存活对象            │
│  2. 整理阶段：将存活对象向一端移动                       │
│  3. 清理边界外的内存                                     │
│                                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │  堆内存：[存活][回收][存活][回收][回收][存活]     │    │
│  │           ↓ 标记并移动                           │    │
│  │           [存活][存活][存活][    ][    ][    ]     │    │
│  │           ↓ 清理边界                             │    │
│  │           [存活][存活][存活][空闲][空闲][空闲]     │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  优点：                                                   │
│  - 无内存碎片                                            │
│  - 适合老年代（对象存活率高）                            │
│                                                          │
│  缺点：                                                   │
│  - 需要移动对象，成本高                                  │
│  - 需要更新引用（对象移动后，引用也要更新）              │
│  - 需要 Stop The World（STW）                            │
│                                                          │
│  应用：老年代 GC（Serial Old、Parallel Old）              │
└─────────────────────────────────────────────────────────┘
```

### 3.4 分代回收理论

```
┌─────────────────────────────────────────────────────────┐
│              JVM 分代回收思想                            │
│                                                          │
│  弱分代假说：                                             │
│  - 绝大多数对象都是"朝生夕死"的                          │
│  - 对象存活时间越长，越不容易死亡                        │
│                                                          │
│  强分代假说：                                             │
│  - 不同存活年龄的对象应该放在不同区域                    │
│  - 不同区域使用不同的 GC 算法                              │
│                                                          │
│  JVM 内存分代：                                           │
│  ┌─────────────────────────────────────────────────┐    │
│  │                  堆 (Heap)                        │    │
│  │  ┌─────────────────┐  ┌─────────────────────┐  │    │
│  │  │    新生代        │  │      老年代          │  │    │
│  │  │  (Young Gen)    │  │   (Old Gen)         │  │    │
│  │  │  ┌─────┬─────┐  │  │                     │  │    │
│  │  │  │Eden │ From│  │  │  长期存活对象        │  │    │
│  │  │  │     │  To │  │  │  大对象直接进入      │  │    │
│  │  │  │     │     │  │  │                     │  │    │
│  │  │  └─────┴─────┘  │  │                     │  │    │
│  │  │  8:1:1 比例     │  │                     │  │    │
│  │  └─────────────────┘  └─────────────────────┘  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  新生代特点：                                             │
│  - 对象存活率低（10% 左右）                               │
│  - 适合复制算法                                          │
│  - Minor GC 频繁（每秒多次）                              │
│                                                          │
│  老年代特点：                                             │
│  - 对象存活率高（50% 以上）                               │
│  - 适合标记整理算法                                      │
│  - Major GC 较少（几秒到几分钟一次）                       │
└─────────────────────────────────────────────────────────┘
```

### 3.5 新生代 GC 详细过程

```
┌─────────────────────────────────────────────────────────┐
│          新生代 Minor GC 过程                             │
│                                                          │
│  初始状态：                                               │
│  ┌─────────────────────────────────────────────────┐    │
│  │  Eden: [Obj1][Obj2][Obj3][Obj4][Obj5]            │    │
│  │  From: [Obj6][Obj7]                              │    │
│  │  To:   [空闲]                                    │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  步骤 1：标记 Eden 和 From 中的存活对象                     │
│  ┌─────────────────────────────────────────────────┐    │
│  │  Eden: [Obj1✓][Obj2][Obj3✓][Obj4][Obj5✓]         │    │
│  │  From: [Obj6✓][Obj7]                             │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  步骤 2：复制存活对象到 To 区                             │
│  ┌─────────────────────────────────────────────────┐    │
│  │  Eden: [Obj1✓][Obj2][Obj3✓][Obj4][Obj5✓]         │    │
│  │  From: [Obj6✓][Obj7]                             │    │
│  │  To:   [Obj1][Obj3][Obj5][Obj6][Obj7]            │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  步骤 3：清空 Eden 和 From                               │
│  ┌─────────────────────────────────────────────────┐    │
│  │  Eden: [空闲]                                    │    │
│  │  From: [空闲]                                    │    │
│  │  To:   [Obj1][Obj3][Obj5][Obj6][Obj7]            │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  步骤 4：交换 From/To 角色                               │
│  ┌─────────────────────────────────────────────────┐    │
│  │  Eden: [空闲]                                    │    │
│  │  From: [Obj1][Obj3][Obj5][Obj6][Obj7] (原 To 区)  │    │
│  │  To:   [空闲] (原 From 区)                        │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  对象年龄增加 1，达到阈值（15）晋升老年代                   │
└─────────────────────────────────────────────────────────┘
```

### 3.6 对象晋升老年代的规则

```
┌─────────────────────────────────────────────────────────┐
│          对象晋升老年代的条件                            │
│                                                          │
│  1. 年龄达到阈值                                          │
│     - 每次 Minor GC，存活对象年龄 +1                       │
│     - 默认阈值：15（-XX:MaxTenuringThreshold）            │
│     - 达到阈值后，下次 GC 时晋升到老年代                    │
│                                                          │
│  2. 大对象直接进入老年代                                 │
│     - 大对象：长数组、大字符串                           │
│     - 参数：-XX:PretenureSizeThreshold                   │
│     - 原因：避免大对象在新生代频繁复制                   │
│                                                          │
│  3. 动态年龄判断                                          │
│     - 同年龄对象总大小 > Survivor 区 50%                  │
│     - 该年龄及更大年龄的对象直接晋升老年代                │
│     - 避免 Survivor 区溢出                                │
│                                                          │
│  4. Survivor 区溢出                                        │
│     - Survivor 区放不下存活对象                          │
│     - 溢出对象直接进入老年代                            │
│                                                          │
│  5. 老年代空间担保                                       │
│     - Minor GC 前检查老年代剩余空间                        │
│     - 老年代剩余 < 历史平均/最大晋升大小                 │
│     - 触发 Full GC 清理老年代                              │
└─────────────────────────────────────────────────────────┘
```

## 四、代码示例

### 4.1 对象分配和 GC 演示

```java
import java.util.*;

public class GCDemo {

    private static final int MB = 1024 * 1024;

    public static void main(String[] args) {
        // 1. 新生代对象分配
        List<byte[]> youngGenList = new ArrayList<>();

        // 分配小对象（在 Eden 区）
        for (int i = 0; i < 100; i++) {
            youngGenList.add(new byte[100 * 1024]);  // 100KB
        }

        // 2. 大对象直接进入老年代
        byte[] bigObject = new byte[5 * MB];  // 5MB

        // 3. 长期存活对象
        List<byte[]> oldGenList = new ArrayList<>();
        for (int i = 0; i < 10; i++) {
            oldGenList.add(new byte[1 * MB]);
        }

        // 4. 制造 GC
        youngGenList.clear();  // 新生代对象可回收
        System.gc();  // 建议 GC（不保证执行）

        System.out.println("GC 完成");
    }
}

// 运行参数：
// -Xms10m -Xmx10m -Xmn2m
// -XX:+PrintGCDetails -XX:+PrintGCDateStamps
```

import java.util.*;

public class ObjectAgeDemo {

    private static final int MB = 1024 * 1024;
    private static List<byte[]> list = new ArrayList<>();

        for (int i = 0; i < 20; i++) {

## 五、常见面试题扩展

### 扩展 1：Minor GC、Major GC、Full GC 的区别？

**答：**

| GC 类型 | 回收区域 | 触发条件 | 停顿时间 |
|--------|----------|----------|----------|
| **Minor GC** | 新生代 | Eden 区满 | 短（几毫秒） |
| **Major GC** | 老年代 | 老年代满 | 较长（几十到几百毫秒） |
| **Full GC** | 堆 + 方法区 | 老年代满、System.gc()、方法区满 | 长（几百毫秒到几秒） |
| **Mixed GC** | 新生代 + 部分老年代 | G1 触发 | 中等 |

**触发条件：**
```
Minor GC 触发：
- Eden 区空间不足
- 新对象无法分配

Major GC 触发：
- 老年代空间不足
- 大对象直接进入老年代，空间不足

Full GC 触发：
- 老年代空间不足
- 方法区/元空间不足
- System.gc() 调用（建议）
- Minor GC 前空间担保失败
```

### 扩展 2：什么是 STW（Stop The World）？

**答：**

```
STW（Stop The World）：
- GC 期间，Java 应用线程暂停
- 只有 GC 线程在工作
- 目的是保证 GC 期间对象状态不变

STW 发生在哪些阶段：
1. 标记阶段（需要对象状态不变）
2. 整理阶段（需要移动对象）
3. 复制阶段（需要移动对象）

哪些 GC 有 STW：
- Serial：STW 时间最长
- Parallel：STW 时间较短（多线程并行）
- CMS：标记和清理阶段 STW 很短
- G1：可预测停顿时间

减少 STW 的方法：
1. 使用并发 GC（CMS、G1）
2. 增加新生代大小（减少 Minor GC 频率）
3. 对象尽量在新生代死亡（减少晋升）
4. 调整 GC 参数（并行线程数等）
```

### 扩展 3：什么是空间担保机制？

**答：**

```
空间担保（Handle Promotion）：

场景：
- Minor GC 前，需要确保老年代有足够空间
- 防止新生代存活对象无法晋升

检查逻辑：
1. 检查老年代剩余空间
2. 计算历史平均晋升大小
3. 如果 老年代剩余 < 历史平均晋升大小
   - 触发 Full GC 清理老年代
4. 如果 Full GC 后空间仍不足
   - OOM

参数：
-XX:+HandlePromotionFailure  # JDK 6u24 后默认开启

流程图：
Minor GC 前
    ↓
检查老年代剩余空间
    ↓
剩余 >= 历史平均？
    ├─> 是：执行 Minor GC
    └─> 否：触发 Full GC
            ↓
        Full GC 后空间足够？
            ├─> 是：执行 Minor GC
            └─> 否：OOM
```

### 扩展 4：如何选择合适的垃圾收集器？

**答：**

```
选择原则：

1. 优先选择 JDK 默认收集器
   - JDK 8：Parallel Scavenge + Parallel Old
   - JDK 11+：G1
   - JDK 17+：G1 或 ZGC

2. 根据应用场景选择

   低延迟场景（Web 服务、实时系统）：
   - CMS（JDK 8）
   - G1（JDK 11+）
   - ZGC（JDK 17+，超低延迟）

   高吞吐量场景（批处理、科学计算）：
   - Parallel Scavenge + Parallel Old
   - 参数：-XX:MaxGCPauseMillis=0

   大堆内存（>32GB）：
   - G1（推荐）
   - ZGC（超低延迟）

3. 调优步骤
   - 先用默认收集器
   - 监控 GC 日志
   - 根据停顿时间/吞吐量调整
   - 必要时更换收集器
```

### 扩展 5：G1 收集器的特点是什么？

**答：**

```
G1（Garbage First）收集器：

特点：
1. 分 region，不分物理分代
   - 堆内存划分为多个 region
   - 每个 region 可以是 Eden、Survivor、Old、Humongous

2. 可预测停顿
   - 设置 -XX:MaxGCPauseMillis=200
   - G1 自动调整回收策略

3. 整体标记，局部复制
   - 全局标记（并发）
   - 优先回收垃圾多的 region（Garbage First）

4. 避免内存碎片
   - 整理过程是复制算法
   - 内存连续

执行步骤：
1. 初始标记（STW）
2. 并发标记
3. 最终标记（STW）
4. 筛选回收（STW）

适用场景：
- 大堆内存（>6GB）
- 低延迟要求
- 多核 CPU

参数：
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
-XX:G1HeapRegionSize=4m
```

## 六、面试答题话术

**面试官问：请说说 GC 算法？**

**标准回答：**

> GC 算法主要包括三个阶段：判断对象存活、标记可回收对象、执行回收。
>
> **判断存活**：Java 使用可达性分析，从 GC Roots 向下遍历，未被引用的对象可回收。
>
> **回收算法**：
> 1. **标记 - 清除**：标记存活对象后清除未标记对象，会产生内存碎片
> 2. **标记 - 复制**：复制存活对象到另一区域后清空原区域，无碎片但内存利用率低
> 3. **标记 - 整理**：标记后将存活对象向一端移动，适合老年代
>
> **分代回收**：新生代用复制算法（Minor GC），老年代用标记整理算法（Major GC/Full GC）。

**加分回答：**

> 补充几点：
> 1. 引用计数法无法解决循环引用，Java 使用可达性分析
> 2. GC Roots 包括：栈中局部变量、静态变量、常量、JNI 引用等
> 3. 新生代 Eden:From:To = 8:1:1，对象年龄达到 15 晋升老年代
> 4. 大对象直接进入老年代，避免在新生代频繁复制
> 5. G1 收集器分 region，可预测停顿时间，适合大堆内存
> 6. STW 发生在标记和整理阶段，并发 GC 可减少 STW 时间
> 7. 空间担保机制防止 Minor GC 时老年代空间不足

## 七、速记表格

| 算法 | 区域 | 优点 | 缺点 |
|------|------|------|------|
| 标记 - 清除 | 老年代 | 简单 | 碎片、效率低 |
| 标记 - 复制 | 新生代 | 高效、无碎片 | 内存利用率低 |
| 标记 - 整理 | 老年代 | 无碎片 | 需移动对象 |

| GC 类型 | 区域 | 算法 | 停顿时间 |
|--------|------|------|----------|
| Minor GC | 新生代 | 复制 | 短 |
| Major GC | 老年代 | 标记整理 | 较长 |
| Full GC | 堆 + 方法区 | 标记整理 | 长 |
| Mixed GC | 新生代 + 部分老年代 | 混合 | 中等（G1） |
