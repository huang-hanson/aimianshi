# [数据库] MySQL 索引的底层结构

## 一、题目分析

### 考察点
- MySQL 索引的数据结构（B+ 树）
- B+ 树与 B 树、二叉树的区别
- 聚簇索引与非聚簇索引
- 覆盖索引、回表查询
- 索引优化原则

### 难度等级
⭐⭐⭐⭐⭐ (5/5 星) - 高频核心难题

### 适用岗位
- 中级/高级 Java 开发工程师
- 数据库相关岗位必考题

---

## 二、核心答案

**一句话回答：**
> MySQL InnoDB 引擎的索引底层采用**B+ 树**数据结构。B+ 树的特点是：**非叶子节点只存索引不存数据**、**叶子节点存所有数据**、**叶子节点之间有指针连接**。InnoDB 有**聚簇索引**（数据与索引在一起）和**非聚簇索引**（二级索引，叶子节点存主键），MyISAM 只有非聚簇索引。

## 三、深度剖析

### 3.1 常见索引数据结构对比

```
┌─────────────────────────────────────────────────────────┐
│          索引数据结构演进                                │
│                                                          │
│  1. 哈希表（Hash）                                       │
│     ┌─────────────────────────────────────────────────┐ │
│     │  优点：等值查询 O(1)                             │ │
│     │  缺点：不支持范围查询、不支持排序               │ │
│     │  应用：Memory 引擎、InnoDB 自适应哈希              │ │
│     └─────────────────────────────────────────────────┘ │
│                                                          │
│  2. 二叉搜索树（BST）                                    │
│     ┌─────────────────────────────────────────────────┐ │
│     │  优点：支持范围查询                             │ │
│     │  缺点：可能退化成链表，树高不平衡               │ │
│     │  应用：无                                       │ │
│     └─────────────────────────────────────────────────┘ │
│                                                          │
│  3. 平衡二叉树（AVL）                                    │
│     ┌─────────────────────────────────────────────────┐ │
│     │  优点：平衡，查询 O(logN)                        │ │
│     │  缺点：树高较高，IO 次数多                         │ │
│     │  应用：无                                       │ │
│     └─────────────────────────────────────────────────┘ │
│                                                          │
│  4. 红黑树（R-B Tree）                                   │
│     ┌─────────────────────────────────────────────────┐ │
│     │  优点：近似平衡，插入删除快                     │ │
│     │  缺点：树高仍然较高                             │ │
│     │  应用：JDK HashMap、TreeMap                     │ │
│     └─────────────────────────────────────────────────┘ │
│                                                          │
│  5. B 树（B-Tree）                                       │
│     ┌─────────────────────────────────────────────────┐ │
│     │  优点：多叉平衡，树高低，IO 少                     │ │
│     │  缺点：非叶子节点也存数据，单页存储索引少       │ │
│     │  应用：MongoDB、部分数据库                      │ │
│     └─────────────────────────────────────────────────┘ │
│                                                          │
│  6. B+ 树（B+ Tree）← MySQL 选择                          │
│     ┌─────────────────────────────────────────────────┐ │
│     │  优点：                                         │ │
│     │  - 非叶子节点只存索引，单页存储索引多，树高更低   │ │
│     │  - 叶子节点存所有数据，查询稳定                  │ │
│     │  - 叶子节点有指针连接，范围查询快               │ │
│     │  缺点：需要回表（二级索引）                     │ │
│     │  应用：MySQL InnoDB、Oracle                     │ │
│     └─────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────┘
```

### 3.2 B+ 树结构详解

```
┌─────────────────────────────────────────────────────────┐
│              B+ 树结构示意图                             │
│                                                          │
│                      根节点                              │
│                  ┌─────────┐                            │
│                  │  17  35  │                            │
│                  │ /  |  \ │                            │
│                  └─┬──┼───┬─┘                            │
│                    │  │   │                              │
│         ┌──────────┘  │   └──────────┐                  │
│         │             │              │                  │
│    ┌─────────┐   ┌─────────┐   ┌─────────┐             │
│    │ 5  12   │   │ 20  28  │   │ 40  50  │  非叶子节点   │
│    │ / |  \  │   │ / |  \  │   │ / |  \  │  (只存索引)   │
│    └─────────┘   └─────────┘   └─────────┘             │
│         │             │              │                  │
│    ┌────┴────┐   ┌────┴────┐   ┌────┴────┐             │
│    ↓         ↓   ↓         ↓   ↓         ↓             │
│ ┌─────┐  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐       │
│ │1,2,3│→ │8,9, │→ │15,16│→ │22,23│→ │38,39│→ │45,46│ ... │
│ │ 数据 │  │ 数据 │ │ 数据 │ │ 数据 │ │ 数据 │ │ 数据 │       │
│ └─────┘  └─────┘ └─────┘ └─────┘ └─────┘ └─────┘       │
│    ↑         ↑   ↑         ↑   ↑         ↑              │
│    └─────────┴───┴─────────┴───┴─────────┘              │
│              叶子节点链表（双向链表）                      │
│              (存所有数据，支持范围查询)                    │
│                                                          │
│  B+ 树特点：                                              │
│  1. 非叶子节点只存索引，不存数据                          │
│  2. 叶子节点存所有数据（整行数据或主键）                  │
│  3. 叶子节点之间有指针连接，形成链表                      │
│  4. 树高度平衡，查询效率稳定 O(logN)                      │
│  5. 范围查询只需找到起点，遍历链表即可                    │
└─────────────────────────────────────────────────────────┘
```

### 3.3 B 树 vs B+ 树对比

```
┌─────────────────────────────────────────────────────────┐
│              B 树 vs B+ 树                               │
│                                                          │
│  B 树结构：                                               │
│  ┌─────────────────────────────────────────────────┐    │
│  │           根节点 [17, 数据]                       │    │
│  │              /    |    \                         │    │
│  │   [5, 数据]      [12, 数据]     [35, 数据]       │    │
│  │    /   \          /   \          /   \           │    │
│  │ [2]  [8]      [10]  [15]    [20]  [50]          │    │
│  │  所有节点都存数据                                  │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  B+ 树结构：                                              │
│  ┌─────────────────────────────────────────────────┐    │
│  │           根节点 [17]                            │    │
│  │              /    |    \                         │    │
│  │   [5]           [12]          [35]              │    │
│  │    ↓             ↓              ↓                │    │
│  │ [2,3,4] → [8,9,10] → [15,16,18] → [40,50,60]    │    │
│  │  只有叶子节点存数据，叶子节点之间有链表            │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  对比表：                                                 │
│  ┌──────────────┬─────────────┬─────────────┐          │
│  │     特性     │    B 树      │    B+ 树     │          │
│  ├──────────────┼─────────────┼─────────────┤          │
│  │ 非叶子节点   │ 存索引 + 数据  │ 只存索引     │          │
│  │ 叶子节点     │ 存数据       │ 存数据 + 链表  │          │
│  │ 查询稳定性   │ 不稳定（可能中途命中）       │ 稳定（都要到叶子）│
│  │ 范围查询     │ 需要中序遍历  │ 遍历链表即可  │          │
│  │ 单页索引数量 │ 少（数据占空间）│ 多（只有索引）│          │
│  │ 树高度       │ 较高         │ 较低         │          │
│  │ IO 次数        │ 较多         │ 较少         │          │
│  │ MySQL 选择    │ ❌          │ ✅          │          │
│  └──────────────┴─────────────┴─────────────┘          │
└─────────────────────────────────────────────────────────┘
```

### 3.4 聚簇索引 vs 非聚簇索引

```
┌─────────────────────────────────────────────────────────┐
│          聚簇索引（Clustered Index）                     │
│                                                          │
│  定义：                                                   │
│  - 数据与索引存储在一起                                  │
│  - 叶子节点存储整行数据                                  │
│  - 一个表只能有一个聚簇索引                              │
│                                                          │
│  InnoDB 聚簇索引：                                       │
│  ┌─────────────────────────────────────────────────┐    │
│  │  主键索引（聚簇索引）                             │    │
│  │  ┌─────────┐                                    │    │
│  │  │  根节点  │                                    │    │
│  │  │  (id)   │                                    │    │
│  │  └────┬────┘                                    │    │
│  │       ↓                                          │    │
│  │  ┌─────────┐                                    │    │
│  │  │ 叶子节点 │                                    │    │
│  │  │ id | name | age | email | ... (整行数据)     │    │
│  │  │ 1  | 张三  | 25  | ...   | ...               │    │
│  │  │ 2  | 李四  | 30  | ...   | ...               │    │
│  │  │ 3  | 王五  | 28  | ...   | ...               │    │
│  │  └─────────┘                                    │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  特点：                                                   │
│  - 主键查询直接定位到数据（无需回表）                    │
│  - 数据按主键顺序存储                                    │
│  - InnoDB 必须有聚簇索引（主键或 rowid）                  │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│          非聚簇索引（Secondary Index）                   │
│                                                          │
│  定义：                                                   │
│  - 索引与数据分离存储                                    │
│  - 叶子节点存储索引列 + 主键值                            │
│  - 一个表可以有多个非聚簇索引                            │
│                                                          │
│  InnoDB 非聚簇索引：                                     │
│  ┌─────────────────────────────────────────────────┐    │
│  │  二级索引（name）                                │    │
│  │  ┌─────────┐                                    │    │
│  │  │  根节点  │                                    │    │
│  │  │ (name)  │                                    │    │
│  │  └────┬────┘                                    │    │
│  │       ↓                                          │    │
│  │  ┌─────────┐                                    │    │
│  │  │ 叶子节点 │                                    │    │
│  │  │ name | id (主键)                             │    │
│  │  │ 张三  | 1   → 回表查聚簇索引                   │    │
│  │  │ 李四  | 2   → 回表查聚簇索引                   │    │
│  │  │ 王五  | 3   → 回表查聚簇索引                   │    │
│  │  └─────────┘                                    │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  特点：                                                   │
│  - 查询需要回表（先查二级索引，再查聚簇索引）            │
│  - 覆盖索引可以避免回表                                  │
│  - 索引过多会影响写性能                                  │
└─────────────────────────────────────────────────────────┘
```

### 3.5 回表查询与覆盖索引

```
┌─────────────────────────────────────────────────────────┐
│          回表查询（Bookmark Lookup）                     │
│                                                          │
│  场景：通过二级索引查询非索引列                           │
│                                                          │
│  示例：                                                   │
│  CREATE TABLE user (                                     │
│      id INT PRIMARY KEY,                                │
│      name VARCHAR(50),                                  │
│      age INT,                                           │
│      email VARCHAR(100),                                │
│      INDEX idx_name (name)                              │
│  );                                                      │
│                                                          │
│  SELECT * FROM user WHERE name = '张三';                │
│                                                          │
│  执行过程：                                               │
│  ┌─────────────────────────────────────────────────┐    │
│  │  1. 查询 idx_name 二级索引                         │    │
│  │     找到 name='张三'，得到 id=1                  │    │
│  │                                                  │    │
│  │  2. 回表查询聚簇索引                              │    │
│  │     根据 id=1，查询主键索引                      │    │
│  │     获取整行数据（name, age, email...）          │    │
│  │                                                  │    │
│  │  3. 返回结果                                     │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  性能影响：                                               │
│  - 每次回表需要额外一次 B+ 树查询                          │
│  - 大量回表会严重影响性能                                │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│          覆盖索引（Covering Index）                      │
│                                                          │
│  定义：                                                   │
│  - 查询的列都在索引中，无需回表                          │
│  - 直接通过二级索引获取所有数据                          │
│                                                          │
│  示例：                                                   │
│  -- 创建联合索引                                         │
│  CREATE INDEX idx_name_age ON user (name, age);         │
│                                                          │
│  -- 覆盖索引查询（不需要回表）                           │
│  SELECT name, age FROM user WHERE name = '张三';        │
│                                                          │
│  执行过程：                                               │
│  ┌─────────────────────────────────────────────────┐    │
│  │  1. 查询 idx_name_age 二级索引                     │    │
│  │     找到 name='张三'，直接获取 age               │    │
│  │     （name 和 age 都在索引中，无需回表）           │    │
│  │                                                  │    │
│  │  2. 返回结果                                     │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  优化效果：                                               │
│  - 避免回表，减少 IO 次数                                 │
│  - 查询性能提升明显                                      │
│  - EXPLAIN 显示 "Using index"                            │
│                                                          │
│  创建覆盖索引的技巧：                                     │
│  - 将查询列加入索引（联合索引）                          │
│  - 避免 SELECT *，只查询需要的列                         │
└─────────────────────────────────────────────────────────┘
```

### 3.6 B+ 树的高度与数据量

```
┌─────────────────────────────────────────────────────────┐
│          B+ 树高度估算                                   │
│                                                          │
│  假设条件（InnoDB 页）：                                  │
│  - 页大小：16KB                                         │
│  - 主键：BIGINT（8 字节）                                │
│  - 指针：6 字节                                          │
│  - 每个索引项：14 字节                                    │
│  - 每页索引数量：16384 / 14 ≈ 1170 个                     │
│                                                          │
│  树高度与数据量：                                         │
│  ┌─────────────────────────────────────────────────┐    │
│  │  高度 1：1170 条数据                              │    │
│  │     根节点                                        │    │
│  │                                                  │    │
│  │  高度 2：1170 × 1170 ≈ 137 万条数据                │    │
│  │     根节点 → 1170 个叶子节点                        │    │
│  │                                                  │    │
│  │  高度 3：1170 × 1170 × 1170 ≈ 16 亿条数据           │    │
│  │     根节点 → 1170 个中间节点 → 137 万个叶子节点     │    │
│  │                                                  │    │
│  │  高度 4：1170^4 ≈ 1.8 万亿条数据                   │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  结论：                                                   │
│  - 16 亿数据只需要 3 层 B+ 树                              │
│  - 每次查询最多 3 次 IO（有索引缓存则更少）               │
│  - 这就是为什么索引查询很快的原因                         │
│                                                          │
│  为什么索引要建在区分度高的列上？                         │
│  - 区分度高 → 过滤的数据多 → 减少回表次数                │
│  - 区分度低（如性别）→ 优化器可能不走索引               │
└─────────────────────────────────────────────────────────┘
```

## 四、代码示例

### 4.1 创建索引示例

```sql
-- 1. 创建表时定义索引
CREATE TABLE user (
    id BIGINT PRIMARY KEY,
    name VARCHAR(50) NOT NULL,
    age INT,
    email VARCHAR(100),
    create_time DATETIME,

    -- 单列索引
    INDEX idx_name (name),

    -- 联合索引（最左前缀原则）
    INDEX idx_name_age (name, age),

    -- 唯一索引
    UNIQUE INDEX idx_email (email)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 2. 已有表添加索引
ALTER TABLE user ADD INDEX idx_create_time (create_time);

-- 3. 创建覆盖索引
CREATE INDEX idx_name_age_email ON user (name, age, email);

-- 4. 查看索引
SHOW INDEX FROM user;

-- 5. 删除索引
DROP INDEX idx_name ON user;
ALTER TABLE user DROP INDEX idx_name_age;
```

### 4.2 覆盖索引优化示例

```sql
-- 原始查询（需要回表）
SELECT name, age, email FROM user WHERE name = '张三';

-- 执行计划
EXPLAIN SELECT name, age, email FROM user WHERE name = '张三';
-- type: ref
-- key: idx_name
-- Extra: Using where  (需要回表)

-- 优化：创建覆盖索引
CREATE INDEX idx_name_age_email ON user (name, age, email);

-- 优化后查询（不需要回表）
EXPLAIN SELECT name, age, email FROM user WHERE name = '张三';
-- type: ref
-- key: idx_name_age_email
-- Extra: Using index  (覆盖索引，不需要回表)
```

### 4.3 最左前缀原则示例

```sql
-- 创建联合索引
CREATE INDEX idx_a_b_c ON table_name (a, b, c);

-- 可以使用索引的查询
SELECT * FROM table_name WHERE a = 1;              -- ✅ 使用索引
SELECT * FROM table_name WHERE a = 1 AND b = 2;    -- ✅ 使用索引
SELECT * FROM table_name WHERE a = 1 AND b = 2 AND c = 3;  -- ✅ 使用索引
SELECT * FROM table_name WHERE a = 1 AND c = 3;    -- ✅ 使用部分索引（只用 a）

-- 不能使用索引的查询
SELECT * FROM table_name WHERE b = 2;              -- ❌ 不使用索引
SELECT * FROM table_name WHERE c = 3;              -- ❌ 不使用索引
SELECT * FROM table_name WHERE b = 2 AND c = 3;    -- ❌ 不使用索引

-- 范围查询后的列不使用索引
SELECT * FROM table_name WHERE a = 1 AND b > 2 AND c = 3;
-- ✅ a 和 b 使用索引，c 不使用（范围查询后失效）
```

### 4.4 索引失效场景示例

```sql
-- 1. 对索引列进行函数操作
SELECT * FROM user WHERE YEAR(create_time) = 2024;  -- ❌ 索引失效
SELECT * FROM user WHERE create_time >= '2024-01-01'
    AND create_time < '2025-01-01';  -- ✅ 使用索引

-- 2. 对索引列进行计算
SELECT * FROM user WHERE age + 1 = 25;  -- ❌ 索引失效
SELECT * FROM user WHERE age = 24;  -- ✅ 使用索引

-- 3. 隐式类型转换
SELECT * FROM user WHERE phone = 13800138000;  -- ❌ 索引失效（phone 是字符串）
SELECT * FROM user WHERE phone = '13800138000';  -- ✅ 使用索引

-- 4. LIKE 以通配符开头
SELECT * FROM user WHERE name LIKE '%张%';  -- ❌ 索引失效
SELECT * FROM user WHERE name LIKE '张%';  -- ✅ 使用索引

-- 5. OR 连接条件
SELECT * FROM user WHERE name = '张三' OR age = 25;
-- ❌ 如果 age 没有索引，全表扫描
SELECT * FROM user WHERE name = '张三' OR email = 'xxx@xxx.com';
-- ✅ 如果 name 和 email 都有索引，索引合并

-- 6. 不等于查询
SELECT * FROM user WHERE age != 25;  -- ❌ 可能不使用索引
SELECT * FROM user WHERE age > 25;  -- ✅ 使用索引

-- 7. IS NULL / IS NOT NULL
SELECT * FROM user WHERE age IS NULL;  -- ⚠️ 可能使用索引（区分度高时）
SELECT * FROM user WHERE age IS NOT NULL;  -- ❌ 通常不使用索引
```

### 4.5 查看和分析索引

```sql
-- 1. 查看表的索引
SHOW INDEX FROM user;

-- 2. 查看索引使用情况
EXPLAIN SELECT * FROM user WHERE name = '张三';

-- EXPLAIN 输出说明：
-- id: 查询序号
-- select_type: 查询类型（SIMPLE, PRIMARY, SUBQUERY 等）
-- table: 表名
-- type: 访问类型（system > const > eq_ref > ref > range > index > ALL）
-- possible_keys: 可能使用的索引
-- key: 实际使用的索引
-- key_len: 索引使用长度
-- ref: 显示哪个字段与索引比较
-- rows: 预计扫描行数
-- Extra: 额外信息（Using index, Using where, Using temporary 等）

-- 3. 查看索引统计信息
SELECT
    TABLE_NAME,
    INDEX_NAME,
    SEQ_IN_INDEX,
    COLUMN_NAME,
    CARDINALITY,  -- 索引区分度
    SUB_PART,     -- 前缀索引长度
    NULLABLE
FROM INFORMATION_SCHEMA.STATISTICS
WHERE TABLE_SCHEMA = 'your_database'
AND TABLE_NAME = 'user';

-- 4. 分析表（更新索引统计信息）
ANALYZE TABLE user;

-- 5. 查看索引大小
SELECT
    TABLE_NAME,
    INDEX_NAME,
    ROUND((INDEX_LENGTH) / 1024 / 1024, 2) AS 'Index Size (MB)'
FROM INFORMATION_SCHEMA.TABLES
WHERE TABLE_SCHEMA = 'your_database'
AND TABLE_NAME = 'user';
```

## 五、常见面试题扩展

### 扩展 1：为什么 MySQL 选择 B+ 树而不是 B 树？

**答：**

```
B+ 树相比 B 树的优势：

1. 非叶子节点只存索引
   - B 树：非叶子节点存索引 + 数据，占用空间大
   - B+ 树：非叶子节点只存索引，单页存储索引更多
   - 结果：B+ 树树高更低，IO 次数更少

2. 查询效率稳定
   - B 树：查询可能在非叶子节点命中，不稳定
   - B+ 树：查询都要到叶子节点，效率稳定

3. 范围查询友好
   - B 树：需要中序遍历，效率低
   - B+ 树：叶子节点有链表，遍历即可

4. 更适合数据库场景
   - 数据库主要是范围查询和排序
   - B+ 树的链表结构天然支持

举例：
- 16KB 页，B 树每页可能存 100 个索引（数据占空间）
- B+ 树每页可以存 1170 个索引（只有索引）
- 同样 16 亿数据，B 树可能 4 层，B+ 树只要 3 层
```

### 扩展 2：聚簇索引和非聚簇索引的区别？

**答：**

| 对比项 | 聚簇索引 | 非聚簇索引 |
|--------|----------|------------|
| **存储方式** | 数据与索引在一起 | 索引与数据分离 |
| **叶子节点** | 存整行数据 | 存索引列 + 主键 |
| **数量限制** | 一个表只能有一个 | 可以有多个 |
| **查询方式** | 直接获取数据 | 需要回表（除非覆盖索引） |
| **适用场景** | 主键查询 | 条件查询、覆盖索引 |
| **InnoDB** | 主键索引 | 二级索引 |
| **MyISAM** | 无 | 所有索引都是非聚簇 |

### 扩展 3：什么是索引下推（ICP）？

**答：**

```
索引下推（Index Condition Pushdown, ICP）：

MySQL 5.6+ 引入的优化技术

场景：
SELECT * FROM user WHERE name LIKE '张%' AND age = 25;
索引：idx_name_age (name, age)

没有 ICP（MySQL 5.6 之前）：
1. 通过 idx_name 找到 name LIKE '张%' 的记录
2. 回表查询聚簇索引
3. 在 Server 层过滤 age = 25

有 ICP（MySQL 5.6+）：
1. 通过 idx_name 找到 name LIKE '张%' 的记录
2. 在索引层直接过滤 age = 25（索引下推）
3. 只回表符合条件的记录

优化效果：
- 减少回表次数
- 减少 IO 操作
- EXPLAIN 显示 "Using index condition"
```

### 扩展 4：如何优化慢查询？

**答：**

```
慢查询优化步骤：

1. 开启慢查询日志
   SET GLOBAL slow_query_log = 'ON';
   SET GLOBAL long_query_time = 1;  # 超过 1 秒的查询

2. 分析慢查询
   mysqldumpslow / pt-query-digest

3. 使用 EXPLAIN 分析
   EXPLAIN SELECT ...
   关注：type、key、rows、Extra

4. 优化索引
   - 添加缺失的索引
   - 删除冗余索引
   - 优化联合索引顺序

5. 优化 SQL
   - 避免 SELECT *
   - 避免函数操作索引列
   - 避免隐式类型转换
   - 优化分页查询

6. 优化表结构
   - 垂直分表（大字段拆分）
   - 水平分表（数据量大）
   - 使用合适的数据类型

7. 其他优化
   - 查询缓存（MySQL 8.0 已移除）
   - 读写分离
   - 分库分表
```

### 扩展 5：什么情况下索引会失效？

**答：**

```
索引失效场景总结：

1. 最左前缀不匹配
   - 联合索引 (a,b,c)，查询 WHERE b=1

2. 对索引列进行函数/计算操作
   - WHERE YEAR(create_time) = 2024
   - WHERE age + 1 = 25

3. 隐式类型转换
   - 字符串字段不加引号

4. LIKE 以通配符开头
   - WHERE name LIKE '%张%'

5. OR 连接条件
   - 只要有一个字段没有索引，全表扫描

6. 不等于查询
   - WHERE age != 25（可能失效）

7. IS NOT NULL
   - 通常不使用索引

8. 数据区分度低
   - 性别、状态等字段
   - 优化器可能选择全表扫描

9. 表数据量小
   - 优化器可能选择全表扫描

10. 自增主键查询
    - WHERE id > 1000000（可能全表扫描）
```

## 六、面试答题话术

**面试官问：请说说 MySQL 索引的底层结构？**

**标准回答：**

> MySQL InnoDB 引擎的索引底层采用**B+ 树**数据结构。
>
> **B+ 树的特点**：
> 1. 非叶子节点只存索引，不存数据
> 2. 叶子节点存所有数据
> 3. 叶子节点之间有指针连接，形成链表
> 4. 树高度平衡，查询效率稳定 O(logN)
>
> **InnoDB 有两种索引**：
> 1. **聚簇索引**：数据与索引存储在一起，叶子节点存整行数据，一个表只能有一个
> 2. **非聚簇索引**（二级索引）：叶子节点存索引列 + 主键值，查询需要回表
>
> **覆盖索引**可以避免回表，提升查询性能。

**加分回答：**

> 补充几点：
> 1. B+ 树相比 B 树，非叶子节点只存索引，单页存储索引更多，树高更低，IO 更少
> 2. B+ 树叶子节点有链表，范围查询只需遍历链表，不需要中序遍历
> 3. 16 亿数据只需要 3 层 B+ 树（每页 1170 个索引）
> 4. 最左前缀原则：联合索引 (a,b,c)，查询必须从 a 开始才能使用索引
> 5. 索引失效场景：函数操作、隐式转换、LIKE 通配符开头、OR 连接等
> 6. 索引下推（ICP）可以在索引层过滤数据，减少回表次数
> 7. 优化慢查询的关键是减少回表，使用覆盖索引

| 叶子节点 | 整行数据 | 索引列 + 主键 |
