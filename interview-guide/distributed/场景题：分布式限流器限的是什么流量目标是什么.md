# 场景题：分布式限流器，限的是什么的流量？目标是什么？

## 一、核心结论

**限的是什么流量：** 限的是**进入系统的请求流量**，更准确地说，是**单位时间内到达的请求数量（QPS/TPS）**。

**目标是什么：** **保护系统不被突发流量打垮，保证核心业务可用，防止雪崩效应。**

---

## 二、限的是什么流量？（三个维度）

### 维度 1：从流量来源分

| 限流对象 | 说明 | 典型场景 |
| :--- | :--- | :--- |
| **用户维度** | 限制单个用户的请求频率 | 防止恶意刷接口、爬虫 |
| **IP 维度** | 限制单个 IP 的请求频率 | 防止 DDoS 攻击、恶意扫描 |
| **接口维度** | 限制单个接口的请求频率 | 保护慢接口、核心接口 |
| **服务维度** | 限制整个服务的总流量 | 保护下游依赖、防止雪崩 |
| **集群维度** | 限制整个集群的总流量 | 控制整体负载、成本优化 |

**生产环境示例：**

```java
// 用户维度：单个用户每秒最多 10 次请求
@RateLimit(key = "user:${userId}", qps = 10)

// IP 维度：单个 IP 每秒最多 100 次请求
@RateLimit(key = "ip:${clientIp}", qps = 100)

// 接口维度：下单接口每秒最多 1000 次请求
@RateLimit(key = "api:createOrder", qps = 1000)

// 服务维度：整个订单服务每秒最多 5000 次请求
@RateLimit(key = "service:order", qps = 5000)
```

---

### 维度 2：从请求类型分

| 请求类型 | 限流策略 | 说明 |
| :--- | :--- | :--- |
| **读请求（GET）** | 宽松限流 | 读操作通常可以放更多流量 |
| **写请求（POST/PUT/DELETE）** | 严格限流 | 写操作涉及数据变更，需要严格限制 |
| **核心接口** | 优先保障 | 如下单、支付，保证有足够配额 |
| **非核心接口** | 可降级限流 | 如评论、点赞，流量大时可限流 |

**生产环境示例：**

```
电商系统：
- 浏览商品（读）：QPS 限制 10000
- 加入购物车（写）：QPS 限制 5000
- 下单（核心写）：QPS 限制 1000
- 评论（非核心）：QPS 限制 500
```

---

### 维度 3：从时间粒度分

| 时间粒度 | 说明 | 典型场景 |
| :--- | :--- | :--- |
| **秒级限流** | 限制每秒请求数 | 最常用，应对突发流量 |
| **分钟级限流** | 限制每分钟请求数 | 平滑短期峰值 |
| **小时级/天级限流** | 限制每小时/每天请求数 | API 调用配额管理 |
| **滑动窗口限流** | 限制任意时间窗口内的请求数 | 更精确的流量控制 |

---

## 三、限流的目标是什么？（核心考点）

### 目标 1：保护系统稳定性（最重要）

**问题场景：**
```
双 11 零点，100 万用户同时点击下单
如果没有没限流：
- 所有请求涌入数据库
- 数据库连接池耗尽
- CPU 100%，内存爆满
- 系统雪崩，全部不可用
```

**限流后：**
```
限流器允许 QPS = 5000
- 每秒只放行 5000 个请求
- 多余请求排队或快速失败
- 数据库负载在可控范围内
- 系统虽然慢，但不会崩
```

**核心思想：** **宁可快速失败，也不要雪崩。**

---

### 目标 2：保证核心业务可用

**问题场景：**
```
系统总容量：10000 QPS
业务 A（核心）：下单，应该保证 5000 QPS
业务 B（非核心）：评论，配额 3000 QPS
业务 C（边缘）：日志，配额 2000 QPS

如果没有限流：
- 业务 C 突发 8000 QPS
- 业务 A 和 B 被挤占，核心业务不可用
```

**限流后：**
```
- 业务 A 配额 5000 QPS，优先保障
- 业务 B 配额 3000 QPS
- 业务 C 配额 2000 QPS，超出部分直接拒绝
- 核心业务永远有资源
```

**核心思想：** **资源隔离，保证核心业务永远有配额。**

---

### 目标 3：防止雪崩效应

**雪崩链路：**
```
服务 A → 服务 B → 服务 C → 数据库

如果服务 C 挂了：
- 服务 B 调用服务 C 超时，线程池耗尽
- 服务 A 调用服务 B 超时，线程池耗尽
- 整个链路全部雪崩
```

**限流器作用：**
```
在服务 B 调用服务 C 的地方加限流：
- 服务 C 挂了，服务 B 限流快速失败
- 服务 B 线程池不耗尽，还能服务其他请求
- 服务 A 不受影响，雪崩被阻断
```

**核心思想：** **下游挂了，上游限流自保，防止连锁反应。**

---

### 目标 4：公平性保障

**问题场景：**
```
100 个用户访问系统
如果没有没限流：
- 5 个恶意用户每秒发 1000 次请求
- 95 个正常用户只能抢到少量资源
- 体验极差
```

**限流后：**
```
单用户限流 QPS = 10
- 每个用户最多 10 QPS
- 恶意用户无法独占资源
- 95 个正常用户公平使用
```

**核心思想：** **防止少数用户独占资源，保证大多数用户的公平性。**

---

### 目标 5：成本控制

**问题场景：**
```
云服务按调用量计费：
- 短信服务：0.05 元/条
- 地图 API：0.01 元/次
- OCR 服务：0.1 元/次

如果没有限流：
- 被恶意调用，一天消耗 10 万元
- 成本失控
```

**限流后：**
```
- 短信服务：每天最多 10 万次（5000 元）
- 地图 API：每天最多 50 万次（5000 元）
- 成本可控，超出部分拒绝
```

**核心思想：** **通过限流控制第三方服务调用量，控制成本。**

## 四、分布式限流 vs 单机限流

### 单机限流（Guava RateLimiter）

```java
// 问题：只能限制单台机器的流量
RateLimiter limiter = RateLimiter.create(100);

// 集群有 10 台机器，总 QPS = 100 × 10 = 1000
// 无法精确控制整个集群的流量
```

### 分布式限流（Redis + Lua）

```lua
-- 核心思想：用 Redis 原子操作，控制整个集群的流量
local key = "rate_limit:api:createOrder"
local limit = 1000  -- 整个集群每秒 1000 次
local current = redis.call("INCR", key)
if current == 1 then
    redis.call("EXPIRE", key, 1)
end
if current > limit then
    return 0  -- 拒绝
else
    return 1  -- 放行
end
```

**核心区别：**

| 对比维度 | 单机限流 | 分布式限流 |
| :--- | :--- | :--- |
| **限流范围** | 单台机器 | 整个集群 |
| **精度** | 粗略（N 台机器 × 单机配额） | 精确（统一配额） |
| **性能** | 高（内存操作） | 较低（网络 IO） |
| **适用场景** | 内部服务、精度要求不高 | 对外 API、核心接口 |

## 五、常用限流算法

### 1. 固定窗口计数器

```
原理：每秒重置计数器，超过阈值就拒绝

[0s─────1s─────2s─────3s]
  |100 次 |50 次  |200 次 |
  ✅      ✅      ❌ (超过 100)

问题：临界问题（0.9s 100 次 + 1.1s 100 次 = 200 次/秒）
```

### 2. 滑动窗口计数器

```
原理：把窗口分成小格，每格独立计数

[0.0s─0.5s─1.0s─1.5s─2.0s]
  |50 次 |50 次 |100 次 |

解决：固定窗口的临界问题
```

### 3. 漏桶算法（Leaky Bucket）

```
原理：固定速率出水，任意速率进水，满了溢出

    进水（任意速率）
       ↓
    ┌───────┐
    │  桶   │ → 出水（固定速率）
    └───────┘

特点：强制固定速率，平滑流量
适用：保护下游系统
```

### 4. 令牌桶算法（Token Bucket）⭐最常用

```
原理：固定速率生产令牌，请求需要消耗令牌

    令牌生产（固定速率）
       ↓
    ┌───────┐
    │  桶   │ → 请求消耗令牌
    └───────┘

特点：允许突发（有令牌就能过）
适用：大多数场景
```

## 六、生产环境限流方案

### 方案 1：Nginx 限流（网关层）

```nginx
# 限流配置
limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

server {
    location /api/ {
        limit_req zone=api_limit burst=20 nodelay;
        proxy_pass http://backend;
    }
}
```

**适用场景：** 入口流量控制、IP 维度限流

---

### 方案 2：Redis + Lua（应用层）

```java
// 伪代码
public boolean tryAcquire(String key, int limit) {
    String script = """
        local current = redis.call("INCR", KEYS[1])
        if current == 1 then
            redis.call("EXPIRE", KEYS[1], 1)
        end
        return current <= ARGV[1] and 1 or 0
    """;
    return (Long) redisTemplate.execute(script, keys, limit);
}
```

**适用场景：** 分布式限流、精确控制

---

### 方案 3：Sentinel（阿里开源）

```java
@SentinelResource(value = "createOrder",
    blockHandler = "handleBlock")
public Order createOrder(OrderDTO dto) {
    // 业务逻辑
}

// 限流处理
public Order handleBlock(OrderDTO dto, BlockException ex) {
    throw new BusinessException("系统繁忙，请稍后再试");
}
```

**适用场景：** 复杂限流规则、熔断降级

## 七、核心对比表

| 对比维度 | 限流前 | 限流后 |
| :--- | :--- | :--- |
| **系统稳定性** | 突发流量可能打垮系统 | 负载可控，不会雪崩 |
| **核心业务** | 可能被非核心业务挤占 | 配额保障，永远可用 |
| **用户体验** | 少数人独占，大多数人慢 | 公平分配，整体可用 |
| **成本控制** | 第三方调用可能失控 | 调用量可控，成本可控 |
| **错误处理** | 超时、重试、雪崩 | 快速失败，优雅降级 |

## 八、面试答题话术（直接背）

**面试官问：分布式限流器，限的是什么的流量？目标是什么？**

答：我从两方面回答：

**第一，限的是什么流量？**

限的是**单位时间内进入系统的请求数量（QPS/TPS）**，具体可以从三个维度理解：

1. **从来源维度**：可以限制单用户、单 IP、单接口、单服务、整个集群的流量。比如单用户每秒最多 10 次请求，防止恶意刷接口；下单接口每秒最多 1000 次，保护核心接口。

2. **从请求类型维度**：读请求宽松限流，写请求严格限流；核心接口优先保障，非核心接口可降级限流。

3. **从时间粒度维度**：秒级限流应对突发流量，分钟级平滑短期峰值，天级控制 API 调用配额。

**第二，目标是什么？**

核心目标有 5 个：

1. **保护系统稳定性**：防止突发流量打垮系统，宁可快速失败，也不要雪崩。

2. **保证核心业务可用**：通过资源隔离，确保下单、支付等核心业务永远有配额。

3. **防止雪崩效应**：下游挂了，上游限流自保，防止连锁反应。

4. **公平性保障**：防止少数用户独占资源，保证大多数用户的公平性。

5. **成本控制**：控制第三方服务调用量，防止成本失控。

**举个例子**：双 11 零点，100 万用户同时下单，如果没有限流，所有请求涌入数据库，连接池耗尽，系统雪崩。有了限流器，每秒只放行 5000 个请求，多余请求快速失败，数据库负载可控，系统虽然慢但不会崩，核心业务始终可用。
