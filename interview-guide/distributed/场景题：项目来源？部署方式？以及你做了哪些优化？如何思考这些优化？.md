## 三、做了哪些优化？（核心考点）

### 优化 1：数据库查询优化 ⭐⭐⭐

#### 问题背景

```
优化前：
- 订单列表查询平均耗时：800ms
- 慢查询日志：每天 10 万 +
- 数据库 CPU 长期 80%+
- 用户投诉：订单页面加载慢
```

#### 优化措施

**1. 索引优化**

```sql
-- 优化前：没有索引，全表扫描
SELECT * FROM orders
WHERE user_id = 123 AND status = 1
ORDER BY create_time DESC
LIMIT 20;

-- 优化后：创建复合索引
CREATE INDEX idx_user_status_create
ON orders(user_id, status, create_time);

-- 查询优化：从 800ms 降到 50ms
EXPLAIN 显示：type=ref, rows=20, Extra=Using index
```

**2. 分页优化（深分页问题）**

```sql
-- 优化前：深分页，扫描 10 万行
SELECT * FROM orders
ORDER BY id
LIMIT 100000, 20;

-- 优化后：覆盖索引 + 子查询
SELECT o.*
FROM orders o
INNER JOIN (
    SELECT id FROM orders
    ORDER BY id
    LIMIT 100000, 20
) tmp ON o.id = tmp.id;

-- 查询优化：从 5s 降到 200ms
```

**3. 读写分离**

```java
// 配置多数据源
@Configuration
public class DataSourceConfig {

    @Bean
    @ConfigurationProperties("spring.datasource.master")
    public DataSource masterDataSource() {
        return DataSourceBuilder.create().build();
    }

    @Bean
    @ConfigurationProperties("spring.datasource.slave")
    public DataSource slaveDataSource() {
        return DataSourceBuilder.create().build();
    }
}

// 路由配置
public class DataSourceRouter extends AbstractRoutingDataSource {
    @Override
    protected Object determineCurrentLookupKey() {
        // 读操作走从库，写操作走主库
        return DBContextHolder.isWrite() ? "master" : "slave";
    }
}
```

#### 优化效果

| 指标 | 优化前 | 优化后 | 提升 |
| :--- | :--- | :--- | :--- |
| 订单列表查询 | 800ms | 50ms | 16 倍 |
| 深分页查询 | 5s | 200ms | 25 倍 |
| 数据库 CPU | 80%+ | 30% | 60% 下降 |
| 慢查询数量 | 10 万/天 | 100/天 | 99.9% 下降 |

### 优化 2：缓存优化 ⭐⭐⭐

#### 问题背景

```
优化前：
- 缓存命中率：30%
- 热点数据（如商品信息）每次查询都打数据库
- 缓存雪崩：大量 key 同时过期，数据库压力大
```

#### 优化措施

**1. 多级缓存架构**

```
用户请求
    │
    ▼
┌─────────────┐
│  本地缓存   │  ← Guava Cache (1 万条，过期时间 1 分钟)
└──────┬──────┘
       │ 未命中
       ▼
┌─────────────┐
│  Redis 缓存  │  ← 热点数据 (100 万条，过期时间 10 分钟)
└──────┬──────┘
       │ 未命中
       ▼
┌─────────────┐
│   数据库    │  ← 全量数据
└─────────────┘
```

**2. 缓存预热**

```java
// 定时任务预热热点数据
@Scheduled(cron = "0 0 2 * * ?")  // 每天凌晨 2 点
public void cacheWarmUp() {
    // 查询 Top 1000 热销商品
    List<Product> hotProducts = productMapper.selectTop1000();

    // 预热到 Redis
    for (Product product : hotProducts) {
        String key = "product:" + product.getId();
        redisTemplate.opsForValue().set(key, product, 10, TimeUnit.MINUTES);
    }

    log.info("缓存预热完成，共{}条数据", hotProducts.size());
}
```

**3. 缓存过期时间随机化（防雪崩）**

```java
// 优化前：固定过期时间，同时过期
redisTemplate.opsForValue().set(key, data, 10, TimeUnit.MINUTES);

// 优化后：随机过期时间，避免同时过期
int expireTime = 600 + new Random().nextInt(120);  // 600-720 秒
redisTemplate.opsForValue().set(key, data, expireTime, TimeUnit.SECONDS);
```

### 优化 3：异步解耦优化 ⭐⭐

#### 问题背景

```
优化前：
- 下单流程同步调用，链路太长
- 下单 → 扣库存 → 发优惠券 → 发积分 → 发短信
- 总耗时：2 秒+
- 任何一个环节失败，整个下单失败
```

#### 优化措施

**1. 核心流程异步化**

```java
// 优化前：同步调用，耗时长
@Transactional
public Order createOrder(OrderDTO dto) {
    // 1. 创建订单
    Order order = orderMapper.insert(dto);

    // 2. 扣库存（200ms）
    inventoryService.decrease(order.getId());

    // 3. 发优惠券（300ms）
    couponService.send(order.getUserId());

    // 4. 发积分（100ms）
    pointsService.add(order.getUserId());

    // 5. 发短信（500ms）
    smsService.send(order.getPhone());

    return order;
    // 总耗时：2 秒+
}

// 优化后：核心流程同步，非核心异步
@Transactional
public Order createOrder(OrderDTO dto) {
    // 1. 创建订单（核心）
    Order order = orderMapper.insert(dto);

    // 2. 扣库存（核心）
    inventoryService.decrease(order.getId());

    // 3. 发送消息，异步处理非核心流程
    OrderMessage msg = new OrderMessage(order.getId());
    rocketMQTemplate.send("order_created", msg);

    return order;
    // 总耗时：300ms
}

// 异步消费者
@RocketMQMessageListener(topic = "order_created", consumerGroup = "order_handler")
public class OrderCreatedConsumer implements RocketMQListener<OrderMessage> {

    @Override
    public void onMessage(OrderMessage msg) {
        // 发优惠券
        couponService.send(msg.getOrderId());

        // 发积分
        pointsService.add(msg.getUserId());

        // 发短信
        smsService.send(msg.getPhone());
    }
}
```

#### 优化效果

| 指标 | 优化前 | 优化后 | 提升 |
| :--- | :--- | :--- | :--- |
| 下单耗时 | 2 秒+ | 300ms | 6 倍 |
| 下单成功率 | 85% | 99% | 14% 提升 |
| 系统吞吐量 | 1000 TPS | 5000 TPS | 5 倍 |

### 优化 4：限流熔断优化 ⭐⭐

#### 问题背景

```
优化前：
- 没有限流，大促时流量洪峰打垮系统
- 没有熔断，下游服务挂了拖垮上游
- 双 11 零点：系统雪崩，5 分钟不可用
```

#### 优化措施

**1. 网关层限流**

```nginx
# Nginx 限流配置
limit_req_zone $binary_remote_addr zone=api_limit:10m rate=100r/s;
limit_req_zone $server_name zone=service_limit:10m rate=1000r/s;

server {
    location /api/ {
        # 单用户限流
        limit_req zone=api_limit burst=50 nodelay;

        # 服务总限流
        limit_req zone=service_limit burst=200 nodelay;

        proxy_pass http://backend;
    }
}
```

**2. 应用层熔断（Sentinel）**

```java
@Service
public class PaymentService {

    @SentinelResource(
        value = "pay",
        blockHandler = "handleBlock",      // 限流处理
        fallback = "handleFallback"        // 降级处理
    )
    public PaymentResult pay(Order order) {
        return paymentApi.pay(order);
    }

    // 限流处理方法
    public PaymentResult handleBlock(Order order, BlockException ex) {
        log.warn("支付限流：{}", order.getId());
        return PaymentResult.busy("系统繁忙，请稍后再试");
    }

    // 降级处理方法
    public PaymentResult handleFallback(Order order, Throwable ex) {
        log.error("支付异常：{}", order.getId(), ex);
        return PaymentResult.fail("支付服务暂时不可用");
    }
}
```

**3. 线程池隔离**

```java
@Configuration
public class ThreadPoolIsolationConfig {

    // 支付服务独立线程池
    @Bean("paymentThreadPool")
    public ThreadPoolExecutor paymentThreadPool() {
        return new ThreadPoolExecutor(
            20, 50, 60, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(100),
            new ThreadPoolExecutor.CallerRunsPolicy()
        );
    }

    // 库存服务独立线程池
    @Bean("inventoryThreadPool")
    public ThreadPoolExecutor inventoryThreadPool() {
        return new ThreadPoolExecutor(
            10, 30, 60, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(50)
        );
    }
}
```

| 指标 | 优化前 | 优化后 | 提升 |
| :--- | :--- | :--- | :--- |
| 大促雪崩次数 | 每次大促都雪崩 | 0 次 | 完全避免 |
| 系统可用性 | 95% | 99.99% | 4 个 9 |
| 故障恢复时间 | 30 分钟 | 1 分钟 | 30 倍 |

### 优化 5：JVM 优化 ⭐

#### 问题背景

```
优化前：
- 频繁 Full GC，每分钟 1 次
- GC 停顿时间：500ms+
- OOM 错误：每周 1-2 次
```

#### 优化措施

**1. 堆内存调整**

```bash
# 优化前：默认配置，堆内存太小
-Xms512m -Xmx512m

# 优化后：根据实际负载调整
-Xms2g -Xmx2g              # 堆内存 2G
-XX:NewRatio=2            # 新生代：老生代 = 1:2
-XX:SurvivorRatio=8       # Eden:Survivor = 8:1:1
-XX:+UseG1GC              # 使用 G1 垃圾收集器
-XX:MaxGCPauseMillis=200  # 最大 GC 停顿时间 200ms
```

**2. 大对象处理**

```java
// 优化前：大对象直接进入老年代，触发 Full GC
List<Order> allOrders = orderMapper.selectAll();

// 优化后：分批处理，避免大对象
int pageSize = 1000;
int pageNum = 0;
while (true) {
    PageHelper.startPage(pageNum, pageSize);
    List<Order> orders = orderMapper.select();
    if (orders.isEmpty()) break;

    // 处理当前批次
    processBatch(orders);
    pageNum++;
}
```

#### 优化效果

| 指标 | 优化前 | 优化后 | 提升 |
| :--- | :--- | :--- | :--- |
| Full GC 频率 | 1 次/分钟 | 1 次/小时 | 60 倍下降 |
| GC 停顿时间 | 500ms+ | 50ms | 10 倍 |
| OOM 错误 | 每周 1-2 次 | 0 次 | 完全避免 |

## 四、如何思考这些优化？（方法论）

### 优化思考框架

```
         ┌─────────────────┐
         │   发现问题      │  ← 监控告警、用户反馈
         └────────┬────────┘
                  │
         ┌────────▼────────┐
         │   分析根因      │  ← 链路追踪、日志分析
         └────────┬────────┘
                  │
         ┌────────▼────────┐
         │   制定方案      │  ← 技术选型、成本评估
         └────────┬────────┘
                  │
         ┌────────▼────────┐
         │   小流量验证    │  ← 灰度发布、A/B 测试
         └────────┬────────┘
                  │
         ┌────────▼────────┐
         │   全量推广      │  ← 监控指标、回滚预案
         └────────┬────────┘
                  │
         ┌────────▼────────┐
         │   持续监控      │  ← 效果评估、持续优化
         └─────────────────┘
```

### 优化原则

#### 1. 先分析，后优化

```
不要盲目优化，先找到瓶颈所在：
- 用监控定位问题（Prometheus、SkyWalking）
- 用数据说话（QPS、响应时间、错误率）
- 找到真正的瓶颈（数据库？缓存？网络？）
```

#### 2. 二八原则

```
80% 的性能问题来自 20% 的代码：
- 优先优化高频调用（如首页、订单列表）
- 优先优化慢接口（响应时间 > 1s）
- 优先优化核心链路（下单、支付）
```

#### 3. 成本收益评估

```
优化前评估投入产出比：
- 优化成本：开发时间、测试成本、运维成本
- 优化收益：性能提升多少？用户体验改善多少？
- 优先级：收益高、成本低的优先做
```

#### 4. 可回滚原则

```
任何优化都要有回滚方案：
- 灰度发布：先 10% 流量，没问题再全量
- 监控告警：优化后指标恶化，立即告警
- 回滚预案：5 分钟内能回滚到优化前版本
```

#### 5. 持续优化

```
优化不是一次性的，是持续过程：
- 业务增长后，原来的优化可能失效
- 定期 Review 系统性能
- 建立性能基线，持续监控
```

## 五、面试答题话术（直接背）

**面试官问：项目来源、部署方式，以及你做了哪些优化？如何思考这些优化？**

答：我从三个方面回答：

**第一，项目背景：**

我负责的是电商平台订单系统，日活 500 万，日均订单 100 万，峰值 QPS 10 万+，大促时能达到 50 万+。技术栈是 Spring Cloud 微服务架构，MySQL 主从复制，Redis 缓存，RocketMQ 消息队列，K8s 容器化部署。

**第二，部署方式：**

应用服务用 K8s Deployment，15 个 Pod 分布在 3 个节点上，配置了健康检查和自动扩缩容。数据库是 1 主 2 从，读写分离。Redis 是哨兵模式高可用。RocketMQ 是多 Master 多 Slave 架构。网关层用 Nginx + Keepalived 做高可用负载均衡。

**第三，做的优化：**

主要有 5 个方面：

1. **数据库查询优化**：通过索引优化、深分页优化、读写分离，订单列表查询从 800ms 降到 50ms，慢查询减少 99.9%。

2. **缓存优化**：搭建多级缓存（本地 + Redis），缓存命中率从 30% 提升到 95%，数据库查询量下降 90%。

3. **异步解耦**：将非核心流程（发优惠券、积分、短信）异步化，下单耗时从 2 秒降到 300ms，吞吐量提升 5 倍。

4. **限流熔断**：网关层限流 + 应用层熔断 + 线程池隔离，大促雪崩从每次必现到完全避免，可用性达到 99.99%。

5. **JVM 优化**：调整堆内存、使用 G1 收集器，Full GC 从每分钟 1 次降到每小时 1 次，OOM 错误完全避免。

**第四，如何思考优化：**

我的优化方法论是：

1. **先分析后优化**：用监控和链路追踪定位真正的瓶颈，用数据说话，不盲目优化。

2. **二八原则**：优先优化高频调用、慢接口、核心链路，80% 的精力解决 20% 的关键问题。

3. **成本收益评估**：优化前评估投入产出比，收益高成本低的优先做。

4. **可回滚原则**：任何优化都要灰度发布、有监控告警、有回滚预案。

5. **持续优化**：优化不是一次性的，建立性能基线，持续监控，随业务增长不断调整。

**举个例子**：我们当时订单列表查询慢，先用 SkyWalking 定位到是数据库慢查询，然后 EXPLAIN 分析发现没走索引，创建复合索引后从 800ms 降到 50ms。这就是先分析后优化的典型案例。
