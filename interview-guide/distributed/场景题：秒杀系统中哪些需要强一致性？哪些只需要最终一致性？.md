# 场景题：秒杀系统中哪些需要强一致性？哪些只需要最终一致性？

## 一、核心结论

**强一致性场景（直接影响资金/资损）：**
- ✅ 库存扣减
- ✅ 用户限购
- ✅ 订单创建
- ✅ 支付状态

**最终一致性场景（可补偿/可容忍延迟）：**
- ✅ 库存同步到 DB
- ✅ 订单详情展示
- ✅ 统计数据（销量、UV、PV）
- ✅ 通知与消息
- ✅ 风控审核

**核心设计原则：** 把强一致的范围缩到最小，其余通过 MQ + 对账 + 补偿实现最终一致

---

## 二、为什么需要区分一致性强弱？

### 秒杀系统的特殊性

```
特点：
├── 瞬时高并发：10 万 + QPS
├── 有限库存：100-1000 件
├── 99% 请求会失败
└── 响应时间要求：1 秒内

矛盾：
├── 全部强一致 → 数据库扛不住，系统崩溃
└── 全部最终一致 → 超卖、资损、用户投诉
```

### CAP 理论的取舍

```
秒杀系统选择：CP + 部分 AP

强一致部分（CP）：
└── 库存、订单、支付 → 保证数据正确

最终一致部分（AP）：
└── 展示、统计、通知 → 保证可用性
```

## 三、需要强一致性的场景

### 3.1 库存扣减 ⭐⭐⭐

**为什么需要强一致：**
```
超卖 = 资损 + 用户投诉 + 平台信誉受损

场景：
- 库存 100 件，10 万人抢购
- 如果并发控制不当，可能卖出 1000 件
- 超卖 900 件 → 赔偿 900 * 商品单价
```

**实现方案：**

```java
// 方案 1：Redis Lua 脚本（推荐，高性能）
@Service
public class StockService {

    @Autowired
    private RedisTemplate redisTemplate;

    /**
     * 预扣库存（原子操作）
     */
    public boolean decreaseStock(Long seckillId) {
        String key = "seckill:stock:" + seckillId;
        
        String script = """
            local stock = tonumber(redis.call('GET', KEYS[1]))
            if stock == nil then
                return -1  -- 库存不存在
            end
            if stock <= 0 then
                return 0   -- 库存不足
            end
            redis.call('DECR', KEYS[1])
            return 1       -- 成功
        """;
        
        Long result = redisTemplate.execute(
            new DefaultRedisScript<>(script, Long.class),
            Collections.singletonList(key)
        );
        
        return result == 1;
    }
}
```

```sql
-- 方案 2：数据库乐观锁（兜底）
UPDATE seckill
SET stock = stock - 1
WHERE id = #{seckillId}
  AND stock > 0;  -- 关键：条件判断

-- 检查影响行数，如果为 0 表示库存不足
```

```
┌─────────────────────────────────────────┐
│  库存扣减流程（强一致）                  │
├─────────────────────────────────────────┤
│  1. Redis Lua 原子预扣                   │
│     └─> 保证不超卖                       │
│                                          │
│  2. 数据库乐观锁兜底                     │
│     └─> 数据持久化                       │
│                                          │
│  3. 失败回滚                            │
│     └─> Redis 库存 +1                    │
└─────────────────────────────────────────┘
```

### 3.2 用户限购 ⭐⭐⭐

**为什么需要强一致：**
```
限购规则：每人限购 1 件

如果限购不强一致：
- 用户 A 并发发起 2 个请求
- 两个请求都通过限购检查
- 用户 A 买了 2 件，突破限购
- 其他用户无法购买，投诉
```

**实现方案：**

```java
@Service
public class PurchaseLimitService {

    @Autowired
    private RedisTemplate redisTemplate;

    /**
     * 检查并设置限购（原子操作）
     */
    public boolean checkAndSetLimit(Long userId, Long seckillId) {
        String key = "seckill:limit:" + userId + ":" + seckillId;
        
        // SETNX 原子操作：如果 key 不存在则设置成功
        Boolean success = redisTemplate.opsForValue()
            .setIfAbsent(key, "1", 24, TimeUnit.HOURS);
        
        return Boolean.TRUE.equals(success);
    }
}
```

```
限购检查流程：
┌──────────┐
│ 用户请求  │
└────┬─────┘
     │
     ▼
┌─────────────────┐
│ Redis SETNX     │ ← 原子操作
│ key: limit:u1:s1│
└────┬────────────┘
     │
     ├─> 成功 → 允许下单
     │
     └─> 失败 → 已购买，拒绝
```

### 3.3 订单创建 ⭐⭐⭐

**为什么需要强一致：**
```
订单必须保证：
├── 订单号全局唯一
├── 订单状态流转正确（待支付→已支付→已完成）
└── 订单与用户/商品关联正确

如果订单不强一致：
├── 重复订单号 → 用户重复付款
├── 状态错乱 → 未支付就发货
└── 关联错误 → A 的订单变成 B 的
```

**实现方案：**

```java
@Service
public class OrderService {

    /**
     * 创建订单（本地事务保证强一致）
     */
    @Transactional(rollbackFor = Exception.class)
    public Order createOrder(Long userId, Long seckillId) {
        // 1. 检查是否重复下单（唯一索引兜底）
        Order exists = orderMapper.getByUserIdAndSeckillId(userId, seckillId);
        if (exists != null) {
            throw new DuplicateOrderException("已下单，请勿重复提交");
        }

        // 2. 生成全局唯一订单号（雪花算法）
        String orderId = snowflake.nextId();

        // 3. 创建订单
        Order order = new Order();
        order.setOrderId(orderId);
        order.setUserId(userId);
        order.setSeckillId(seckillId);
        order.setStatus(OrderStatus.UNPAID);
        orderMapper.insert(order);

        // 4. 扣减库存（必须在同一事务）
        int affected = seckillMapper.decreaseStock(seckillId);
        if (affected == 0) {
            throw new StockNotEnoughException("库存不足");
        }

        return order;
    }
}
```

```sql
-- 数据库唯一索引（兜底）
CREATE UNIQUE INDEX uk_order_id ON orders(order_id);
CREATE UNIQUE INDEX uk_user_seckill ON orders(user_id, seckill_id);
```

### 3.4 支付状态 ⭐⭐⭐

**为什么需要强一致：**
```
支付状态直接影响资金：
├── 未支付标记为已支付 → 资损
├── 已支付标记为未支付 → 用户投诉
└── 重复扣款 → 用户投诉 + 监管处罚
```

**实现方案：**

```java
@Service
public class PaymentService {

    /**
     * 处理支付回调（强一致）
     */
    @Transactional(rollbackFor = Exception.class)
    public void handlePaymentCallback(PaymentCallback callback) {
        // 1. 查询订单
        Order order = orderMapper.getByOrderId(callback.getOrderId());
        if (order == null) {
            throw new OrderNotFoundException("订单不存在");
        }

        // 2. 检查订单状态（防止重复处理）
        if (order.getStatus() == OrderStatus.PAID) {
            log.warn("订单已支付，忽略回调：{}", callback.getOrderId());
            return;
        }

        // 3. 更新支付状态
        Payment payment = new Payment();
        payment.setOrderId(callback.getOrderId());
        payment.setTransactionId(callback.getTransactionId());
        payment.setAmount(callback.getAmount());
        payment.setStatus(PaymentStatus.SUCCESS);
        paymentMapper.insert(payment);

        // 4. 更新订单状态
        order.setStatus(OrderStatus.PAID);
        order.setPaidTime(LocalDateTime.now());
        orderMapper.update(order);
    }
}
```

```
支付状态流转：
┌─────────┐    ┌─────────┐    ┌─────────┐
│ 待支付   │───>│ 已支付   │───>│ 已完成   │
└─────────┘    └─────────┘    └─────────┘
     ↑              ↑
     │              │
  用户下单      支付回调
```

## 四、只需要最终一致性的场景

### 4.1 库存同步到 DB ⭐⭐

**为什么可以最终一致：**
```
秒杀流程：
1. Redis 预扣库存（强一致）
2. 发送 MQ 消息
3. 异步扣减 DB 库存

允许短暂不一致：
- Redis 扣了 100 件
- DB 可能还在 100 件（延迟 1-5 秒）
- 但最终会同步到 0 件

即使 DB 同步失败：
- 可以通过 Redis → DB 对账修复
- 不影响用户购买和资金安全
```

**实现方案：**

```java
// 1. Redis 预扣后发送 MQ
@Service
public class SeckillService {
    
    public boolean seckill(Long userId, Long seckillId) {
        // Redis 预扣
        boolean success = stockService.decreaseStock(seckillId);
        if (!success) {
            return false;
        }
        
        // 发送 MQ，异步同步到 DB
        StockSyncMessage message = new StockSyncMessage(seckillId, userId);
        rocketMQTemplate.send("stock_sync", message);
        
        return true;
    }
}

// 2. 消费者异步扣减 DB
@RocketMQMessageListener(topic = "stock_sync", consumerGroup = "stock_sync_cg")
public class StockSyncConsumer implements RocketMQListener<StockSyncMessage> {
    
    @Override
    public void onMessage(StockSyncMessage message) {
        // 异步扣减 DB 库存（慢一点没关系）
        seckillMapper.decreaseStockDB(message.getSeckillId());
    }
}

// 3. 定时对账（兜底）
@Component
public class StockReconciliation {
    
    @Scheduled(cron = "0 */5 * * * ?")  // 每 5 分钟对账
    public void reconcile() {
        List<Seckill> seckills = seckillMapper.selectAll();
        for (Seckill seckill : seckills) {
            // 获取 Redis 库存
            Long redisStock = redisTemplate.opsForValue()
                .get("seckill:stock:" + seckill.getId());
            
            // 获取 DB 库存
            Integer dbStock = seckillMapper.getStock(seckill.getId());
            
            // 如果不一致，以 Redis 为准修复 DB
            if (!Objects.equals(redisStock, dbStock.longValue())) {
                log.warn("库存不一致：seckillId={}, redis={}, db={}", 
                         seckill.getId(), redisStock, dbStock);
                seckillMapper.fixStock(seckill.getId(), redisStock.intValue());
            }
        }
    }
}
```

### 4.2 订单详情展示 ⭐

**为什么可以最终一致：**
```
用户下单后：
- 列表页/详情页可以短暂显示"处理中"
- 1-5 秒后刷新看到最新状态即可
- 不影响核心交易流程

技术方案：
├── 订单创建后发送 MQ
├── 消费者更新订单缓存
└── 用户轮询/WebSocket 通知
```

**实现方案：**

```java
// 1. 下单后返回"处理中"
@RestController
public class OrderController {
    
    @PostMapping("/order/create")
    public Result createOrder(@RequestBody CreateOrderRequest request) {
        // 异步下单
        orderMQService.send(request);
        
        // 立即返回处理中
        return Result.processing("订单处理中，请稍后查看");
    }
    
    @GetMapping("/order/status")
    public Result getOrderStatus(@RequestParam Long orderId) {
        // 轮询查询状态
        Order order = orderMapper.getById(orderId);
        if (order == null) {
            return Result.processing("处理中");
        }
        return Result.success(order);
    }
}

// 2. 消费者更新缓存
@RocketMQMessageListener(topic = "order_created", consumerGroup = "order_cache_cg")
public class OrderCacheConsumer implements RocketMQListener<Order> {
    
    @Autowired
    private RedisTemplate redisTemplate;
    
    @Override
    public void onMessage(Order order) {
        // 更新订单缓存
        String key = "order:" + order.getId();
        redisTemplate.opsForValue().set(key, order, 7, TimeUnit.DAYS);
    }
}
```

### 4.3 统计数据（销量、UV、PV） ⭐

**为什么可以最终一致：**
```
统计数据特点：
├── 不影响交易流程
├── 用户看到延迟几秒的销量没关系
└── 可以通过异步聚合计算

如果强一致：
├── 每次请求都要 COUNT(*) → 数据库爆炸
└── 响应时间不可控
```

**实现方案：**

```java
// 1. 实时数据写入 Redis（高性能）
@Service
public class StatisticsService {
    
    @Autowired
    private RedisTemplate redisTemplate;
    
    /**
     * 记录销量（Redis 原子操作）
     */
    public void recordSale(Long seckillId) {
        String key = "seckill:stats:" + seckillId;
        redisTemplate.opsForValue().increment(key + ":sales");
        redisTemplate.opsForHyperLogLog().add(key + ":uv", getUserId());
        redisTemplate.opsForValue().increment(key + ":pv");
    }
}

// 2. 异步聚合到 DB（最终一致）
@RocketMQMessageListener(topic = "sale_recorded", consumerGroup = "stats_aggregate_cg")
public class StatisticsConsumer implements RocketMQListener<SaleRecord> {
    
    @Override
    public void onMessage(SaleRecord record) {
        // 聚合统计（慢一点没关系）
        statisticsMapper.accumulateSales(record.getSeckillId(), record.getCount());
    }
}

// 3. 定时全量对账（兜底）
@Component
public class StatisticsReconciliation {
    
    @Scheduled(cron = "0 0 */1 * * ?")  // 每小时对账
    public void reconcile() {
        // 从订单表重新计算销量
        List<Statistics> stats = orderMapper.calculateStatistics();
        for (Statistics stat : stats) {
            statisticsMapper.updateStatistics(stat);
        }
    }
}
```

### 4.4 通知与消息 ⭐

**为什么可以最终一致：**
```
通知类型：
├── 短信通知
├── APP 推送
├── 站内信
└── 邮件

可以延迟：
- 晚几秒收到通知没关系
- 即使发送失败，可以重试
- 不影响交易本身
```

**实现方案：**

```java
// 1. 发送 MQ 消息
@Service
public class OrderService {
    
    public void createOrder(Long userId, Long seckillId) {
        // 创建订单...
        
        // 发送通知消息
        NotificationMessage msg = new NotificationMessage(userId, orderId);
        rocketMQTemplate.send("notification", msg);
    }
}

// 2. 消费者异步发送
@RocketMQMessageListener(topic = "notification", consumerGroup = "notification_cg")
public class NotificationConsumer implements RocketMQListener<NotificationMessage> {
    
    @Autowired
    private SmsService smsService;
    
    @Autowired
    private PushService pushService;
    
    @Override
    public void onMessage(NotificationMessage message) {
        try {
            // 发送短信
            smsService.send(message.getUserId(), "秒杀成功！");
            
            // 发送推送
            pushService.send(message.getUserId(), "订单已创建");
        } catch (Exception e) {
            // 重试或记录死信
            log.error("发送通知失败", e);
            throw e;  // 触发重试
        }
    }
}
```

### 4.5 风控审核 ⭐⭐

**为什么可以最终一致：**
```
风控场景：
├── 检测黄牛/刷单
├── 检测异常 IP/设备
└── 检测批量注册

处理方式：
- 先允许下单（保证用户体验）
- 异步风控审核
- 发现异常再取消订单/拦截发货
```

**实现方案：**

```java
// 1. 下单后发送风控审核
@Service
public class OrderService {
    
    public void createOrder(Long userId, Long seckillId) {
        // 创建订单...
        
        // 发送风控审核
        RiskCheckMessage msg = new RiskCheckMessage(userId, orderId);
        rocketMQTemplate.send("risk_check", msg);
    }
}

// 2. 异步风控审核
@RocketMQMessageListener(topic = "risk_check", consumerGroup = "risk_cg")
public class RiskCheckConsumer implements RocketMQListener<RiskCheckMessage> {
    
    @Autowired
    private RiskCheckService riskCheckService;
    
    @Override
    public void onMessage(RiskCheckMessage message) {
        // 风控检查
        RiskResult result = riskCheckService.check(message.getUserId());
        
        if (result.isRisk()) {
            // 发现风险，取消订单
            orderService.cancelOrder(message.getOrderId(), "风控拦截");
            
            // 加入黑名单
            blacklistService.add(message.getUserId());
        }
    }
}
```

## 五、一致性强弱对比表

| 场景 | 一致性要求 | 原因 | 实现方案 | 不一致的后果 |
|------|-----------|------|----------|-------------|
| **库存扣减** | 强一致 | 超卖=资损 | Redis Lua + DB 乐观锁 | 超卖、赔偿 |
| **用户限购** | 强一致 | 突破公平原则 | Redis SETNX | 用户投诉 |
| **订单创建** | 强一致 | 交易核心数据 | 本地事务 + 唯一索引 | 重复订单、资金错误 |
| **支付状态** | 强一致 | 直接涉及资金 | 本地事务 + 幂等 | 重复扣款、资损 |
| **库存同步 DB** | 最终一致 | 可补偿 | MQ 异步 + 对账 | 短暂不一致，可修复 |
| **订单展示** | 最终一致 | 不影响交易 | 缓存 + 轮询 | 用户体验稍差 |
| **统计数据** | 最终一致 | 非核心 | Redis + 异步聚合 | 数据延迟 |
| **通知消息** | 最终一致 | 可重试 | MQ 异步 | 延迟收到 |
| **风控审核** | 最终一致 | 事后补救 | MQ 异步 + 取消订单 | 可事后拦截 |

## 七、面试答题话术

**面试官问：秒杀系统中哪些需要强一致性？哪些只需要最终一致性？**

**答：** 我从三个方面回答：

**第一，判断标准：**
我判断一致性强弱的标准是**是否直接影响资金和资损**。直接影响交易的环节必须强一致，可以补偿或延迟的环节可以最终一致。

**第二，强一致的场景有 4 个：**

1. **库存扣减**：超卖会直接导致资损和用户投诉，必须用 Redis Lua 脚本或数据库乐观锁保证原子性。

2. **用户限购**：突破限购会损害其他用户利益，用 Redis SETNX 原子操作保证一人一单。

3. **订单创建**：订单号唯一性、订单状态流转必须正确，用本地事务 + 唯一索引保证。

4. **支付状态**：直接涉及资金，重复扣款或状态错乱会导致资损，用本地事务 + 幂等性保证。

**第三，最终一致的场景有 5 个：**

1. **库存同步到 DB**：Redis 扣完后，异步同步到 DB，允许几秒延迟，通过定时对账兜底。

2. **订单详情展示**：用户下单后可以显示"处理中"，轮询或 WebSocket 通知最新状态。

3. **统计数据**：销量、UV、PV 等可以延迟，用 Redis 计数 + 异步聚合。

4. **通知消息**：短信、推送可以延迟发送，用 MQ 削峰异步。

5. **风控审核**：先让用户下单，异步风控检查，发现异常再取消订单。

**核心设计原则：** 把强一致的范围缩到最小，其余通过 MQ + 对账 + 补偿实现最终一致。这样既能抗住 10 万 + QPS，又能保证数据正确性。

## 八、扩展问题

### 扩展 1：Redis 和 DB 库存不一致怎么办？

**答：** 三种方案：
1. **定时对账**：每 5 分钟对比 Redis 和 DB 库存，以 Redis 为准修复 DB
2. **消息队列**：每次 Redis 扣减都发送 MQ，消费者更新 DB
3. **监听 Binlog**：用 Canal 监听 DB Binlog，反向更新 Redis（不推荐，复杂）

### 扩展 2：如何保证 MQ 消息不丢失？

**答：** 三个环节保证：
1. **生产端**：发送确认 + 失败重试 + 本地消息表
2. **Broker 端**：持久化到磁盘 + 多副本同步
3. **消费端**：手动 ACK + 消费失败重试 + 死信队列

### 扩展 3：对账发现不一致如何处理？

**答：** 三种策略：
1. **以 Redis 为准**：Redis 是实时数据，DB 同步延迟，用 Redis 修复 DB
2. **以 DB 为准**：如果对 Redis 数据不放心，用 DB 修复 Redis
3. **人工介入**：差异过大时告警，人工排查
