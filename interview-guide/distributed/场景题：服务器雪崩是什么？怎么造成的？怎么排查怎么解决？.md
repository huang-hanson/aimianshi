# 场景题：服务器雪崩是什么？怎么造成的？怎么排查？怎么解决？

## 一、核心结论

**服务器雪崩是什么：** 指**系统中某个服务或组件故障，引发连锁反应，导致整个系统像雪崩一样全面瘫痪**。

**核心特征：** 单点故障 → 连锁反应 → 系统全面崩溃

**排查思路：** 从监控告警 → 链路追踪 → 日志分析 → 定位根因

**解决方案：** 限流、熔断、降级、隔离、冗余、自动故障转移

---

## 二、服务器雪崩是什么？（图解）

### 雪崩效应示意图

```
                    用户请求
                       │
                       ▼
                  ┌─────────┐
                  │  网关层  │
                  └────┬────┘
                       │
         ┌─────────────┼─────────────┐
         ▼             ▼             ▼
   ┌─────────┐   ┌─────────┐   ┌─────────┐
   │ 服务 A   │   │ 服务 B   │   │ 服务 C   │ ← 服务 C 挂了！
   └────┬────┘   └────┬────┘   └────┬────┘
        │             │             │
        └─────────────┼─────────────┘
                      ▼
                 ┌─────────┐
                 │ 数据库   │
                 └─────────┘

雪崩过程：
1. 服务 C 挂了（比如响应慢、OOM）
2. 调用服务 C 的服务 B 线程池耗尽
3. 调用服务 B 的服务 A 线程池耗尽
4. 网关层请求全部超时
5. 整个系统瘫痪！
```

### 真实案例

```
某电商系统双 11 经历：
00:00  活动开始，流量洪峰
00:01  优惠券服务响应变慢（50ms → 2s）
00:02  订单服务调用优惠券服务超时，线程池耗尽
00:03  支付服务调用订单服务超时，线程池耗尽
00:04  网关层请求大量超时，用户无法下单
00:05  整个系统雪崩，活动失败
```

## 三、服务器雪崩是怎么造成的？（7 大原因）

### 原因 1：流量洪峰超出系统承载能力 ⭐⭐⭐

**典型场景：**
- 双 11、618 等大促活动
- 突发热点事件（微博热搜）
- 恶意攻击（DDoS）

**雪崩过程：**
```
正常 QPS：1000
系统承载能力：5000 QPS
活动开始 QPS：50000

结果：
- CPU 100%，内存爆满
- 请求处理越来越慢
- 线程池耗尽
- 系统崩溃
```

---

### 原因 2：服务依赖超时拖垮线程池 ⭐⭐⭐

**典型场景：**
```
服务 A → 服务 B → 服务 C

服务 C 响应慢（从 50ms 变成 5s）
↓
服务 B 调用服务 C 超时，线程阻塞
↓
服务 B 线程池耗尽（200 个线程全阻塞）
↓
服务 A 调用服务 B 超时，线程阻塞
↓
服务 A 线程池耗尽
↓
整个链路雪崩
```

**核心问题：** 没有设置超时时间 + 没有熔断机制

---

### 原因 3：数据库连接池耗尽 ⭐⭐

**典型场景：**
```
正常情况：
- 数据库连接池：100 个连接
- 活跃连接：20 个
- 请求响应：50ms

慢 SQL 出现后：
- 某查询全表扫描，响应从 50ms 变成 5s
- 活跃连接飙升到 100 个
- 新请求等待连接，排队超时
- 应用层线程池也耗尽
- 雪崩！
```

---

### 原因 4：缓存雪崩/穿透/击穿 ⭐⭐⭐

**缓存雪崩：**
```
大量缓存 key 同时过期
↓
所有请求直接打到数据库
↓
数据库瞬间压力过大
↓
数据库崩溃
```

**缓存穿透：**
```
查询不存在的数据（如 id = -1）
↓
缓存不命中，请求直达数据库
↓
恶意攻击者大量查询不存在的数据
↓
数据库被拖垮
```

**缓存击穿：**
```
热点 key 过期瞬间
↓
大量请求同时查询这个 key
↓
数据库瞬间压力过大
↓
数据库崩溃
```

---

### 原因 5：内存泄漏/OOM ⭐⭐

**典型场景：**
```java
// 内存泄漏示例
public static Map<String, Object> cache = new HashMap<>();

public void processData(String key, Object data) {
    cache.put(key, data);  // 只增不减，最终 OOM
}

// 运行几天后：
// heap 内存满 → Full GC → 还是满 → OOM
// 服务挂掉
```

**雪崩过程：**
```
服务 A OOM 挂掉
↓
负载均衡把流量转发到服务 B、C
↓
服务 B、C 流量翻倍，也 OOM 了
↓
所有实例都挂了
↓
雪崩！
```

---

### 原因 6：线程池配置不当 ⭐

**典型场景：**
```java
// 错误配置：线程池太小
ExecutorService executor = new ThreadPoolExecutor(
    10,   // 核心线程数太小
    20,   // 最大线程数太小
    60, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(100)  // 队列太小
);

// 正常 QPS 500，突然来了一波 2000 QPS
// 线程池瞬间满了，请求全部拒绝
// 调用方超时，连锁反应
```

---

### 原因 7：网络故障/带宽打满 ⭐

**典型场景：**
```
正常带宽使用：100Mbps / 1Gbps = 10%

突发情况：
- 某个服务大量返回大数据
- 或者被 DDoS 攻击
- 带宽瞬间打满到 100%

结果：
- 其他服务请求发不出去
- 超时、重试
- 雪崩！
```

### 第一步：看监控告警（1 分钟内）

**监控面板核心指标：**

| 指标 | 正常值 | 告警值 | 说明 |
| :--- | :--- | :--- | :--- |
| **QPS** | 1000 | > 5000 | 流量是否异常 |
| **响应时间** | < 100ms | > 1s | 服务是否变慢 |
| **错误率** | < 0.1% | > 1% | 服务是否异常 |
| **CPU 使用率** | < 50% | > 80% | 资源是否充足 |
| **内存使用率** | < 70% | > 90% | 是否内存泄漏 |
| **线程池活跃数** | < 50% | > 90% | 线程池是否耗尽 |
| **数据库连接数** | < 50% | > 90% | 连接池是否耗尽 |

**常用监控工具：**
- Prometheus + Grafana（资源监控）
- SkyWalking/Pinpoint（链路追踪）
- ELK（日志分析）
- Zabbix（基础监控）

---

### 第二步：看链路追踪（5 分钟内）

**SkyWalking 链路追踪示例：**

```
请求链路：网关 → 订单服务 → 支付服务 → 数据库

正常情况：
网关 (5ms) → 订单 (50ms) → 支付 (30ms) → 数据库 (10ms) = 95ms

雪崩时：
网关 (5ms) → 订单 (50ms) → 支付 (3000ms) → 数据库 (超时) = 3100ms+
                                       ↑
                                   问题在这里！
```

**快速定位：** 看哪个环节的耗时突然变长，问题就在那里。

**Linux 命令排查：**

**典型问题定位：**

| :--- | :--- |

| 内存 100% | 内存泄漏、大对象、缓存过多 |
| 磁盘 IO 100% | 大量读写、慢 SQL、日志暴增 |
| 带宽 100% | DDoS 攻击、大数据传输 |
| 连接数爆满 | 连接池耗尽、连接泄漏 |

## 五、服务器雪崩怎么解决？（核心考点）

### 解决方案架构图

```
                    用户请求
                       │
                       ▼
              ┌─────────────────┐
              │   限流层        │  ← 第一道防线：限流
              │  (Nginx/Sentinel)│    挡住超出承载的流量
              └────────┬────────┘
                       │
                       ▼
              ┌─────────────────┐
              │   熔断层        │  ← 第二道防线：熔断
              │  (Hystrix/Sentinel)│  快速失败，防止连锁反应
              └────────┬────────┘
                       │
                       ▼
              ┌─────────────────┐
              │   降级层        │  ← 第三道防线：降级
              │  (返回默认值)    │    保证核心功能可用
              └────────┬────────┘
                       │
                       ▼
              ┌─────────────────┐
              │   隔离层        │  ← 第四道防线：隔离
              │  (线程池隔离)    │    防止线程池耗尽
              └────────┬────────┘
                       │
                       ▼
                  业务服务
```

---

### 方案 1：限流（挡住超出承载的流量）⭐⭐⭐

**限流位置：**

| 位置 | 方案 | 说明 |
| :--- | :--- | :--- |
| **网关层** | Nginx 限流 | 最外层，挡住恶意请求 |
| **应用层** | Sentinel/Guava | 精确控制，支持复杂规则 |
| **分布式限流** | Redis + Lua | 控制整个集群流量 |

**Nginx 限流配置：**
```nginx
# 限制每个 IP 每秒 10 个请求
limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

server {
    location /api/ {
        limit_req zone=api_limit burst=20 nodelay;
        proxy_pass http://backend;
    }
}
```

**Sentinel 限流配置：**
```java
@SentinelResource(value = "createOrder",
    blockHandler = "handleBlock")
public Order createOrder(OrderDTO dto) {
    // 业务逻辑
}

// 限流处理方法
public Order handleBlock(OrderDTO dto, BlockException ex) {
    throw new BusinessException("系统繁忙，请稍后再试");
}
```

---

### 方案 2：熔断（快速失败，防止连锁反应）⭐⭐⭐

**熔断原理：**
```
         关闭（正常）
           │
           │ 失败率 > 50%
           ▼
         打开（熔断）───────→ 直接返回，不调用服务
           │
           │ 5 秒后
           ▼
         半开（试探）
           │
           │ 成功 → 关闭
           │ 失败 → 打开
```

**Sentinel 熔断配置：**
```java
@SentinelResource(value = "callPaymentService",
    fallback = "paymentFallback",           // 降级方法
    blockHandler = "paymentBlockHandler")   // 限流处理方法
public PaymentResult callPaymentService(Order order) {
    return paymentService.pay(order);
}

// 熔断降级方法
public PaymentResult paymentFallback(Order order, Throwable ex) {
    log.error("支付服务熔断", ex);
    return PaymentResult.fail("支付服务繁忙，请稍后重试");
}
```

---

### 方案 3：降级（保证核心功能可用）⭐⭐

**降级策略：**

| 降级类型 | 说明 | 示例 |
| :--- | :--- | :--- |
| **返回默认值** | 返回缓存/默认数据 | 推荐服务挂了，返回热门商品 |
| **返回错误提示** | 友好提示用户稍后重试 | "系统繁忙，请稍后再试" |
| **关闭非核心功能** | 保核心，弃边缘 | 关闭评论、点赞，保下单 |
| **异步处理** | 先接收请求，后续处理 | 先下单，后续再扣库存 |

**降级代码示例：**
```java
@Service
public class RecommendService {

    @Autowired
    private RedisTemplate redisTemplate;

    // 推荐服务降级：返回缓存的热门商品
    public List<Product> getRecommendProducts(String userId) {
        try {
            // 调用推荐服务
            return recommendApi.getRecommend(userId);
        } catch (Exception e) {
            // 降级：返回缓存的热门商品
            log.error("推荐服务异常，降级到热门商品", e);
            return (List<Product>) redisTemplate.opsForValue().get("hot_products");
        }
    }
}
```

### 方案 4：隔离（防止线程池耗尽）⭐⭐

**线程池隔离：**
```java
// 为每个依赖服务配置独立线程池
@Configuration
public class ThreadPoolConfig {

    // 支付服务线程池
    @Bean("paymentThreadPool")
    public ThreadPoolExecutor paymentThreadPool() {
        return new ThreadPoolExecutor(
            20, 50, 60, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(100)
        );
    }

    // 用户服务线程池
    @Bean("userThreadPool")
    public ThreadPoolExecutor userThreadPool() {
        return new ThreadPoolExecutor(
            10, 30, 60, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(50)
        );
    }
}

// 使用独立线程池调用
public PaymentResult callPayment(Order order) {
    return paymentThreadPool.submit(() -> paymentService.pay(order)).get();
}
```

**隔离效果：**
```
支付服务挂了 → 支付线程池耗尽 → 不影响用户服务线程池 → 其他功能正常
```

### 方案 5：冗余（提高可用性）⭐

**多实例部署：**
```
           负载均衡
               │
    ┌──────────┼──────────┐
    │          │          │
    ▼          ▼          ▼
 服务 A-1    服务 A-2    服务 A-3

任何一个实例挂了，还有其他实例可以服务
```

**多机房部署：**
```
         用户请求
            │
    ┌───────┼───────┐
    │       │       │
    ▼       ▼       ▼
 北京机房  上海机房  广州机房

任何一个机房挂了，其他机房可以接管
```

---

### 方案 6：自动故障转移（秒级恢复）⭐

**K8s 健康检查 + 自动重启：**
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
spec:
  template:
    spec:
      containers:
      - name: order
        livenessProbe:  # 存活探针
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 10
        readinessProbe:  # 就绪探针
          httpGet:
            path: /ready
            port: 8080
          initialDelaySeconds: 5
          periodSeconds: 5
```

**效果：** 服务挂了 10 秒内自动重启，30 秒内恢复服务。

## 七、面试答题话术（直接背）

**面试官问：服务器雪崩是什么？怎么造成的？怎么排查？怎么解决？**

答：我从四个方面回答：

**第一，什么是服务器雪崩？**

服务器雪崩是指**系统中某个服务或组件故障，引发连锁反应，导致整个系统全面瘫痪**。就像雪崩一样，一小块雪滑落引发大规模雪崩。

**第二，怎么造成的？**

常见原因有 7 个：

1. **流量洪峰**：双 11 等活动流量超出系统承载能力
2. **服务依赖超时**：下游服务响应慢，拖垮上游线程池
3. **数据库连接池耗尽**：慢 SQL 导致连接池满了
4. **缓存雪崩/穿透/击穿**：大量请求直达数据库
5. **内存泄漏/OOM**：服务挂掉，流量转移到其他实例
6. **线程池配置不当**：线程池太小，无法应对突发流量
7. **网络故障/带宽打满**：请求发不出去

**第三，怎么排查？**

五步排查法：

1. **看监控告警**：哪些服务挂了、什么时候开始的、影响范围多大
2. **看链路追踪**：用 SkyWalking 看哪个环节耗时变长
3. **看资源使用**：CPU、内存、磁盘 IO、带宽是否打满
4. **看日志**：错误日志、超时日志、OOM 日志
5. **定位根因**：慢 SQL、代码 bug、配置问题、流量洪峰

**第四，怎么解决？**

六道防线：

1. **限流**：网关层 Nginx 限流，应用层 Sentinel 限流，挡住超出承载的流量
2. **熔断**：下游服务挂了快速失败，防止连锁反应
3. **降级**：返回默认值或缓存数据，保证核心功能可用
4. **隔离**：为每个依赖服务配置独立线程池，防止线程池耗尽
5. **冗余**：多实例、多机房部署，提高可用性
6. **自动故障转移**：K8s 健康检查，服务挂了自动重启

**举个例子**：我们当时双 11 经历了雪崩，优惠券服务响应慢拖垮了订单服务。后来我们加了 Sentinel 限流熔断，配置了线程池隔离，设置了降级返回缓存数据，从此再也没雪崩过。
