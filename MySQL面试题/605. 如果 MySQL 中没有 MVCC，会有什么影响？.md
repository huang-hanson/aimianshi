# 605. 如果 MySQL 中没有 MVCC，会有什么影响？

---

## 一、核心结论

**MVCC 是 MySQL InnoDB 引擎的「核心基石技术」，如果没有 MVCC，InnoDB 会彻底退化为「锁驱动的低并发数据库」，完全无法支撑互联网高并发业务。**

---

## 二、没有 MVCC 的核心影响（分维度精讲）

### 1. 并发性能彻底崩盘：读写完全互斥，高并发场景直接瘫痪

#### 有 MVCC 时的状态

- 读操作（普通 `SELECT`）是**快照读**，基于历史版本，**完全不加锁**

- 写操作（`INSERT/UPDATE/DELETE`）只加行锁，**不阻塞读操作**

- 实现「读不阻塞写，写不阻塞读」，完美适配「读多写少」的互联网高并发场景

#### 没有 MVCC 时的状态

- 所有读操作都必须变成**当前读**，强制加**共享锁（S锁）**

- 写操作加**排他锁（X锁）**，读写、写写完全互斥

- 读操作必须等所有写操作提交，写操作必须等所有读操作释放锁

- 高并发场景下，大量事务排队等待锁，系统吞吐量暴跌 90% 以上，直接瘫痪

---

### 2. 事务隔离级别无法正常实现，一致性彻底失控

#### 有 MVCC 时的状态

- MVCC 是 InnoDB 实现 **RR（可重复读）** 隔离级别的核心技术

- 通过 `undo log` 版本链 + `read view` 一致性视图，在不加锁的前提下，解决脏读、不可重复读，配合间隙锁解决幻读

- 同时支持 RC（读已提交）隔离级别，满足不同业务的一致性需求

#### 没有 MVCC 时的状态

- 只能通过**加锁**来实现事务隔离：

    - 要解决脏读：读操作加锁，等写操作提交

    - 要解决不可重复读：事务全程加锁，不释放

    - 要解决幻读：加表级锁，完全串行执行

- 最终结果：**只能用 SERIALIZABLE（串行化）隔离级别**，所有事务串行执行，并发能力为 0

- 业务系统完全无法使用，只能单线程处理请求

---

### 3. 互联网业务场景完全失效，MySQL 失去核心竞争力

#### 有 MVCC 时的状态

- 完美适配电商、社交、金融等绝大多数互联网业务：

    - 电商商品详情页：高并发读，读不加锁，不影响下单写操作

    - 社交动态流：高并发读，不影响用户发布动态

    - 金融交易：兼顾一致性与并发，支持高并发转账

- MySQL 成为全球最流行的开源关系型数据库，核心就是 MVCC 带来的高并发能力

#### 没有 MVCC 时的状态

- 所有读操作加锁，高并发读场景直接卡死：

    - 电商商品页：用户下单时，所有查商品的请求全部阻塞

    - 社交平台：用户发动态时，所有刷动态的请求全部等待

    - 金融系统：转账时，所有查余额的请求全部排队

- MySQL 彻底失去高并发优势，无法适配互联网业务，只能用于低并发的企业内部系统

---

### 4. 锁冲突与死锁概率暴增，系统稳定性急剧下降

#### 有 MVCC 时的状态

- 读操作不加锁，锁冲突仅发生在写操作之间

- 死锁概率极低，仅在多写操作交叉加锁时出现

- 系统稳定性高，锁等待少，故障恢复快

#### 没有 MVCC 时的状态

- 所有读操作加锁，锁冲突概率呈指数级增长：

    - 读操作和写操作冲突，读操作和读操作不冲突，但大量读锁会阻塞写操作

    - 长事务读操作会长期持有锁，阻塞所有写操作，导致系统雪崩

    - 死锁概率暴增，大量事务因锁等待超时，业务报错频发

- 系统稳定性完全失控，运维成本极高

---

### 5. 其他核心影响（面试加分项）

#### （1）undo log 失去存在意义

- MVCC 是 undo log 的核心使用场景，没有 MVCC，undo log 仅用于事务回滚，功能大幅缩水

- 版本链彻底消失，数据历史版本不再存储，回滚性能也会受影响

#### （2）InnoDB 核心优势丧失

- InnoDB 对比 MyISAM 的核心优势就是「事务支持 + 高并发」，没有 MVCC，InnoDB 仅剩下事务支持，并发性能远不如 MyISAM

- InnoDB 彻底失去存在意义，MySQL 会退化为 MyISAM 主导的非事务型数据库

#### （3）业务代码复杂度暴增

- 没有 MVCC，所有一致性问题都需要业务代码通过加锁、重试、排队来解决

- 业务代码需要处理大量锁冲突、死锁、超时问题，开发成本、维护成本指数级增长

## 三、有 MVCC vs 无 MVCC 核心对比表（面试速记）

| 对比维度         | 有 MVCC（InnoDB 默认）           | 无 MVCC（锁驱动）                    |
| ---------------- | -------------------------------- | ------------------------------------ |
| **读写阻塞**     | 读写不阻塞，读不加锁，写加行锁   | 读写完全互斥，读加共享锁，写加排他锁 |
| **并发性能**     | 高，完美适配高并发场景           | 极低，高并发场景直接瘫痪             |
| **事务隔离**     | 支持 RR/RC，不加锁实现一致性     | 仅支持 SERIALIZABLE，完全串行        |
| **锁冲突概率**   | 低，仅写操作之间冲突             | 极高，读写、写写全部冲突             |
| **死锁概率**     | 低，仅多写操作交叉加锁时出现     | 暴增，大量事务锁等待                 |
| **适用场景**     | 互联网高并发业务（电商、社交等） | 低并发企业内部系统                   |
| **MySQL 竞争力** | 全球最流行开源数据库             | 退化为非事务型数据库，失去优势       |

## 四、面试答题话术（直接背）

**面试官问：如果 MySQL 中没有 MVCC，会有什么影响？**

答：MVCC 是 MySQL InnoDB 引擎的核心基石技术，如果没有 MVCC，会带来以下致命影响：

1. **并发性能彻底崩盘**：所有读操作必须加共享锁，读写完全互斥，高并发场景下大量事务排队，吞吐量暴跌，系统直接瘫痪；

2. **事务隔离级别无法正常实现**：只能通过加锁实现隔离，最终只能用串行化级别，所有事务串行执行，并发能力为 0；

3. **互联网业务场景完全失效**：无法适配电商、社交等读多写少的高并发场景，MySQL 失去核心竞争力；

4. **锁冲突与死锁概率暴增**：大量读锁阻塞写操作，长事务导致系统雪崩，死锁频发，系统稳定性失控；

5. **业务代码复杂度暴增**：所有一致性问题需要业务代码手动处理，开发、维护成本指数级增长。

6. 简单来说，没有 MVCC，InnoDB 就会退化为低并发的锁驱动数据库，完全无法支撑现代互联网业务。
