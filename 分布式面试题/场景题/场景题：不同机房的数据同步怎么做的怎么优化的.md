# 场景题：不同机房的数据同步怎么做的？怎么优化的？

## 一、核心结论

**数据同步方案：** 主流方案是**MySQL 主从复制 + binlog 异步/半同步复制**，或者**MQ 消息队列 + 本地事务表**。

**优化方向：** **降低延迟、保证一致性、避免数据冲突、提高容灾能力**。

---

## 二、为什么需要跨机房数据同步？

### 业务场景

| 场景 | 说明 | 同步要求 |
| :--- | :--- | :--- |
| **异地容灾** | 主机房挂了，备机房能立刻接管 | 数据一致性优先 |
| **就近访问** | 用户访问离自己最近的机房 | 低延迟优先 |
| **数据汇总** | 多个机房数据汇总到总部分析 | 最终一致性即可 |
| **流量分担** | 多个机房同时对外提供服务 | 双向同步，避免冲突 |

### 典型架构

```
                    ┌─────────────┐
                    │   用户请求   │
                    └──────┬──────┘
                           │
         ┌─────────────────┼─────────────────┐
         │                 │                 │
         ▼                 ▼                 ▼
   ┌───────────┐    ┌───────────┐    ┌───────────┐
   │  北京机房  │    │  上海机房  │    │  广州机房  │
   │  Master   │◄──►│  Master   │◄──►│  Master   │
   └───────────┘    └───────────┘    └───────────┘
```

## 三、主流数据同步方案（4 种）

### 方案 1：MySQL 主从复制（最常用）⭐

#### 架构原理

```
┌──────────────┐      binlog       ┌──────────────┐
│  主机房 MySQL │ ────────────────► │  备机房 MySQL │
│   (Master)   │    异步/半同步     │    (Slave)   │
└──────────────┘                   └──────────────┘
```

#### 同步流程

```
1. 主机房写入数据
   └─> 写入 binlog

2. 备机房 IO 线程
   └─> 拉取 binlog 到 relay log

3. 备机房 SQL 线程
   └─> 重放 relay log 中的数据
```

#### 三种复制模式对比

| 模式 | 原理 | 延迟 | 一致性 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **异步复制** | 主库写完就返回，不等待从库 | 最低 | 可能丢数据 | 数据汇总、分析 |
| **半同步复制** | 主库等待至少一个从库确认 | 中等 | 基本不丢数据 | 异地容灾 |
| **全同步复制** | 主库等待所有从库确认 | 最高 | 强一致 | 金融核心数据 |

#### 配置示例

```sql
-- 主机房配置（开启半同步）
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';
SET GLOBAL rpl_semi_sync_master_enabled = ON;
SET GLOBAL rpl_semi_sync_master_timeout = 1000; -- 超时 1 秒切异步

-- 备机房配置
INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';
SET GLOBAL rpl_semi_sync_slave_enabled = ON;
START SLAVE;
```

#### 优缺点

| 优点 | 缺点 |
| :--- | :--- |
| MySQL 原生支持，配置简单 | 跨机房网络延迟高（10-50ms） |
| 数据一致性有保障 | 网络抖动可能导致同步中断 |
| 支持故障自动切换 | 双向同步有冲突风险 |

---

### 方案 2：MQ 消息队列 + 本地事务表（推荐）⭐⭐

#### 架构原理

```
┌──────────────┐      ┌──────────┐      ┌──────────────┐
│  主机房应用  │ ────►│   MQ     │ ────►│  备机房应用  │
│  + 本地事务表 │      │(RocketMQ)│      │  + 数据消费  │
└──────────────┘      └──────────┘      └──────────────┘
```

#### 同步流程（事务消息）

```
1. 主机房写入本地数据 + 发送消息（事务消息）
   └─> 本地事务表插入记录
   └─> 发送 half 消息到 MQ

2. MQ 确认消息接收

3. 提交本地事务
   └─> 提交数据库事务

4. 备机房消费消息
   └─> 写入备机房数据库
   └─> ACK 确认消费
```

#### 核心代码

```java
// 主机房：发送事务消息
@Transactional
public void createOrder(Order order) {
    // 1. 写入本地数据
    orderMapper.insert(order);

    // 2. 发送事务消息
    Message msg = new Message("order_sync", order.toJson());
    rocketMQTemplate.sendMessageInTransaction("txGroup", msg);
}

// 备机房：消费消息
@RocketMQMessageListener(topic = "order_sync", consumerGroup = "sync_group")
public class OrderSyncConsumer implements RocketMQListener<Order> {
    public void onMessage(Order order) {
        // 1. 检查是否已处理（幂等）
        if (syncRecord.exists(order.getId())) {
            return;
        }

        // 2. 写入备机房数据
        orderMapper.insert(order);

        // 3. 记录同步日志
        syncRecord.insert(order.getId());
    }
}
```

#### 优缺点

| 优点 | 缺点 |
| :--- | :--- |
| 解耦，不依赖数据库复制 | 需要自己保证一致性 |
| 支持复杂业务逻辑 | 开发成本高 |
| 天然支持双向同步 | 需要处理消息重复、丢失 |
| 可以过滤、转换数据 | 延迟比数据库复制高 |

---

### 方案 3：Canal 监听 binlog（轻量级）⭐

#### 架构原理

```
┌──────────────┐      ┌──────────┐      ┌──────────────┐
│  主机房 MySQL │ ────►│  Canal   │ ────►│  备机房 MySQL │
│   (binlog)   │      │  监听器   │      │              │
└──────────────┘      └──────────┘      └──────────────┘
```

#### 同步流程

```
1. Canal 伪装成 MySQL Slave
   └─> 连接主机房 MySQL

2. Canal 解析 binlog
   └─> 转换成 JSON/消息格式

3. 发送到 MQ 或直接写入备机房
   └─> 备机房应用数据
```

#### 配置示例

```java
// Canal 监听器
@CanalListener(destination = "example", filter = ".*\\..*")
public class CanalDataSyncListener {

    @CanalHandler(eventType = EventType.INSERT)
    public void onInsert(CanalEntry.RowChange rowChange) {
        // 解析 binlog
        CanalEntry.RowChange rowChange = CanalEntry.RowChange.parseFrom(entry.getStoreValue());

        // 发送到 MQ
        rocketMQTemplate.send("data_sync_topic", rowChange);
    }
}
```

#### 优缺点

| 优点 | 缺点 |
| :--- | :--- |
| 无侵入，不需要改业务代码 | 只能单向同步 |
| 轻量级，部署简单 | 无法保证事务一致性 |
| 支持数据转换、过滤 | binlog 格式变化可能影响解析 |

#### 架构原理

┌─────────────────────────────────────────────┐
│              TiDB 集群                       │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐     │
│  │ 北京节点 │  │ 上海节点 │  │ 广州节点 │     │

| :--- | :--- |

| 强一致性（Raft 协议） | 延迟受网络影响 |
| 自动故障切换 | 需要更换数据库 |
| 对应用透明 | 学习成本高 |

## 四、数据同步优化方案（核心考点）

### 优化 1：降低同步延迟

#### 问题原因

```
跨机房网络延迟 = 物理距离 / 光速 + 网络设备延迟

北京 - 上海：约 1000km
- 理论延迟：1000km / 200000km/s = 5ms
- 实际延迟：10-20ms（经过多个路由器）
```

#### 优化方案

| 方案 | 说明 | 效果 |
| :--- | :--- | :--- |
| **专线连接** | 租用运营商专线，减少网络跳数 | 延迟从 20ms 降到 5ms |
| **压缩传输** | binlog 压缩后传输 | 减少 50% 网络传输量 |
| **批量同步** | 积累一批数据一起同步 | 减少网络往返次数 |
| **并行复制** | 多个线程并行回放 binlog | 回放速度提升 5-10 倍 |

#### MySQL 并行复制配置

```sql
-- 开启基于 WRITESET 的并行复制
SET GLOBAL slave_parallel_type = 'WRITESET';
SET GLOBAL slave_parallel_workers = 16;  -- 16 个并行线程
SET GLOBAL slave_preserve_commit_order = ON;
```

### 优化 2：保证数据一致性

#### 问题场景

```
主机房写入数据后挂了，备机房还没同步过来
→ 数据丢失
→ 用户数据不一致
```

#### 解决方案

| 方案 | 原理 | 适用场景 |
| :--- | :--- | :--- |
| **半同步复制** | 主库等待至少一个从库确认 | 一般容灾 |
| **加权确认** | 等待多个从库确认（权重和≥2） | 高可用要求 |
| **对账补偿** | 定期对比主备数据，自动修复 | 最终一致性 |
| **双写校验** | 同时写两个机房，对比结果 | 金融核心数据 |

#### 对账补偿代码

```java
// 定时对账任务
@Scheduled(cron = "0 0 2 * * ?")  // 每天凌晨 2 点
public void dataReconciliation() {
    // 1. 获取主机房数据 checksum
    Map<String, Long> masterChecksum = orderMapper.getChecksum("2024-01-01");

    // 2. 获取备机房数据 checksum
    Map<String, Long> slaveChecksum = slaveOrderMapper.getChecksum("2024-01-01");

    // 3. 对比差异
    List<String> diffIds = findDifference(masterChecksum, slaveChecksum);

    // 4. 修复差异数据
    for (String id : diffIds) {
        Order order = masterOrderMapper.selectById(id);
        slaveOrderMapper.insertOrUpdate(order);
    }
}
```

### 优化 3：避免数据冲突（双向同步）

#### 冲突场景

```
北京机房：更新用户余额为 1000
上海机房：同时更新用户余额为 2000

双向同步后：
- 北京数据变成 2000
- 上海数据变成 1000
→ 数据不一致！
```

#### 解决方案

| 方案 | 原理 | 优缺点 |
| :--- | :--- | :--- |
| **单元化路由** | 同一用户固定路由到同一机房 | 无冲突，但灵活性差 |
| **时间戳覆盖** | 新时间戳覆盖旧数据 | 简单，但有时钟同步问题 |
| **版本号覆盖** | 高版本号覆盖低版本号 | 可靠，需要额外字段 |
| **优先级覆盖** | 高优先级机房覆盖低优先级 | 简单，可能丢数据 |

#### 单元化路由实现

```java
// 根据用户 ID 路由到固定机房
public int getTargetRoomId(String userId) {
    int hash = Math.abs(userId.hashCode());
    int roomCount = 3; // 北京、上海、广州
    return hash % roomCount;
}

// 路由规则：
// userId % 3 = 0 → 北京机房（写）
// userId % 3 = 1 → 上海机房（写）
// userId % 3 = 2 → 广州机房（写）

// 所有机房的数据双向同步到其他机房
```

#### 版本号覆盖实现

```java
// 表结构增加 version 字段
CREATE TABLE user_balance (
    user_id BIGINT PRIMARY KEY,
    balance DECIMAL(10,2),
    version BIGINT,  -- 版本号
    update_time TIMESTAMP
);

// 更新时检查版本号
UPDATE user_balance
SET balance = 1000, version = version + 1
WHERE user_id = 123 AND version < 5;  -- 只更新版本号更大的

// 如果影响行数 = 0，说明有冲突，跳过或记录日志
```

### 优化 4：提高容灾能力

#### 问题场景

```
主机房挂了，备机房如何快速接管？
```

#### 解决方案

| 方案 | 原理 | 切换时间 |
| :--- | :--- | :--- |
| **VIP 漂移** | 虚拟 IP 从主机房漂到备机房 | 秒级 |
| **DNS 切换** | 修改 DNS 解析到备机房 | 分钟级（有缓存） |
| **配置中心切换** | Nacos/Apollo 切换数据源配置 | 秒级 |
| **自动故障检测** | 心跳检测 + 自动切换 | 秒级 |

#### 配置中心切换示例

```java
// Nacos 配置监听
@NacosConfigListener(dataId = "datasource.json")
public void onDataSourceChange(String config) {
    DataSource newDs = parseDataSource(config);

    // 动态切换数据源
    DynamicDataSourceContextHolder.setMaster(newDs);

    log.info("数据源已切换到新机房");
}

// 故障检测
@Scheduled(fixedRate = 5000)  // 每 5 秒检测一次
public void healthCheck() {
    if (!masterDataSource.healthCheck()) {
        // 主机房挂了，切换到备机房
        failoverToSlave();
    }
}
```

## 五、核心对比表

| 同步方案 | 延迟 | 一致性 | 开发成本 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **MySQL 主从复制** | 中（10-50ms） | 高 | 低 | 异地容灾、数据汇总 |
| **MQ+ 本地事务表** | 低（50-100ms） | 中（最终一致） | 中 | 业务数据同步 |
| **Canal 监听 binlog** | 低（10-50ms） | 中（最终一致） | 低 | 轻量级同步 |
| **TiDB 分布式数据库** | 最低（Raft） | 最高（强一致） | 高 | 核心业务、金融场景 |

## 六、面试答题话术（直接背）

**面试官问：不同机房的数据同步怎么做的？怎么优化的？**

答：我从**同步方案**和**优化方向**两方面回答：

**第一，主流的数据同步方案有 4 种：**

1. **MySQL 主从复制**：最常用，利用 binlog 异步或半同步复制。配置简单，MySQL 原生支持，但跨机房网络延迟较高（10-50ms）。

2. **MQ 消息队列 + 本地事务表**：用 RocketMQ 事务消息，保证本地写入和消息发送的原子性。解耦性好，支持复杂业务，但开发成本高。

3. **Canal 监听 binlog**：轻量级方案，Canal 伪装成 MySQL Slave 解析 binlog，发送到 MQ 或备机房。无侵入，但无法保证事务一致性。

4. **TiDB 分布式数据库**：终极方案，原生支持跨机房同步，Raft 协议保证强一致。但成本高，需要更换数据库。

**第二，优化方向有 4 个：**

1. **降低同步延迟**：用专线连接减少网络跳数（从 20ms 降到 5ms），开启 MySQL 并行复制提升回放速度（16 个线程，提升 5-10 倍）。

2. **保证数据一致性**：用半同步复制保证至少一个从库确认，定期做数据对账，发现差异自动修复。

3. **避免数据冲突**：双向同步时，用单元化路由（同一用户固定路由到同一机房）或版本号覆盖（高版本覆盖低版本）。

4. **提高容灾能力**：通过配置中心（Nacos/Apollo）实现秒级故障切换，心跳检测自动 failover。

**举个例子**：我们当时是主机房写数据，通过 RocketMQ 事务消息同步到备机房，延迟 50ms 左右。优化措施包括：开启半同步复制保证不丢数据，用单元化路由避免双向同步冲突，每天凌晨对账修复差异数据，通过 Nacos 配置中心实现秒级故障切换。
